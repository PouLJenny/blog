Abstract
摘要
The rise of AI coding assistants has reignited interest in an old idea: what if specifications—not code—were the primary artifact of software development? Spec-driven development (SDD) inverts the traditional workflow by treating specifications as the source of truth and code as a generated or verified secondary artifact. This paper provides practitioners with a comprehensive guide to SDD, covering its principles, workflow patterns, and supporting tools. We present three levels of specification rigor—spec-first, spec-anchored, and spec-as-source—with clear guidance on when each applies. Through analysis of tools ranging from Behavior-Driven Development frameworks to modern AI-assisted toolkits like GitHub Spec Kit, we demonstrate how the spec-first philosophy maps to real implementations. We present case studies from API development, enterprise systems, and embedded software, illustrating how different domains apply SDD. We conclude with a decision framework helping practitioners determine when SDD provides value and when simpler approaches suffice.
AI 编程助手的兴起让一个老话题重新受到关注：如果把"规约"——而不是代码——当作软件开发的核心产物，会怎样？规约驱动开发（Spec-Driven Development, SDD）颠倒了传统工作流：它把规约视为"事实之源（source of truth）"，把代码视为由规约生成或验证而得的次级产物。本文为从业者提供一份关于 SDD 的完整指南，涵盖其原则、工作流模式以及配套工具。我们提出三种严格程度不同的规约层级——规约先行（spec-first）、规约锚定（spec-anchored）、规约即源（spec-as-source）——并清晰说明各自适用的场景。通过分析从行为驱动开发（BDD）框架到 GitHub Spec Kit 等现代 AI 辅助工具包的一系列工具，我们展示"规约先行"理念如何落地到真实工程实现。我们给出了来自 API 开发、企业系统和嵌入式软件的案例研究，说明不同领域如何应用 SDD。最后，我们给出一套决策框架，帮助从业者判断何时 SDD 能带来价值，何时用更简单的方法就已足够。

I. Introduction
一、引言
For decades, code has been the king of software development. Requirements documents exist, but they drift. Design diagrams are drawn, but they rot. Tests are written, but often after the fact. The code—whatever it actually does—becomes the de facto truth of the system.
几十年来，代码一直是软件开发中的"王者"。需求文档确实存在，但会漂移；设计图也会画，但会腐烂；测试会写，但常常是事后补的。最后，代码——不管它实际在做什么——事实上成了整个系统的"真相"。
This code-centric reality has consequences. When a new developer asks "what should this function do?" the answer is often "read the code." When a stakeholder asks "does the system meet our requirements?" the answer requires reverse-engineering intent from implementation. When an AI coding assistant is asked to add a feature, it must guess what the developer wants from a vague prompt.
这种"以代码为中心"的现实会带来后果。当新来的开发者问"这个函数应该做什么？"，答案往往是"去读代码"。当利益相关方问"系统是否满足我们的需求？"，回答这个问题就需要从实现中反推意图。当 AI 编程助手被要求加一个功能时，它必须从一句模糊的提示里去猜开发者想要什么。
Spec-driven development (SDD) [2][4] offers an alternative: make the specification the source of truth, and let code derive from it. Instead of coding first and documenting later (or never), teams write clear specifications of intended behavior, then generate, implement, or verify code against those specifications. The spec becomes the authoritative description that both humans and machines use to understand, build, and maintain the system.
规约驱动开发（SDD）[2][4] 给出了另一种选择：让规约成为事实之源，让代码由它派生。不再是"先写代码、之后补文档（或者根本不补）"，而是：团队先把预期行为以清晰的规约写出来，再针对这份规约去生成、实现或验证代码。规约成了人和机器都依赖的权威描述，用来理解、构建和维护系统。
A. The AI Catalyst
A. AI 的催化作用
While spec-first thinking is not new—Test-Driven Development (TDD) and Behavior-Driven Development (BDD) have advocated for it for years—the emergence of AI coding assistants [15][28] has made SDD newly relevant. The problem is simple: AI models are excellent at pattern completion but poor at mind reading.
"规约先行"的思路并不新——测试驱动开发（TDD）和行为驱动开发（BDD）多年来一直在倡导这一点——但 AI 编程助手 [15][28] 的出现让 SDD 重新变得紧迫。问题很简单：AI 模型非常擅长模式补全，却不擅长读心。
Consider a developer who prompts an AI with: "Add photo sharing to my app." The AI must guess: What format? What permissions model? What size limits? Cloud storage or local? Compression? The result is often plausible-looking code that makes dozens of unstated assumptions—many of them wrong.
设想一个开发者对 AI 说："给我的应用加一个照片分享功能。"AI 必须去猜：什么格式？什么权限模型？尺寸上限多少？云端存储还是本地？要不要压缩？结果常常是一段"看起来合理"的代码，背后却塞进了几十条未说明的假设——其中很多是错的。
This is what practitioners call "vibe coding"—relying on loose prompts that lead to inconsistent or erroneous outputs from LLMs. By providing AI with unambiguous, executable contracts, SDD enhances the reliability of coding agents and opens new avenues for scalable software creation.
这就是业界所说的"凭感觉编程（vibe coding）"——依赖松散的提示，让大模型输出前后不一致甚至错误的结果。SDD 给 AI 提供无歧义、可执行的契约，既提升了编程智能体的可靠性，也为"可规模化地生产软件"打开了新的路径。
Now consider the same request with a specification: "Users can upload JPEG or PNG photos up to 10MB. Photos are stored in S3 with user-ID-prefixed keys. Only the uploader can delete their photos. Photos are resized to 1024px max dimension on upload." The AI now has enough information to generate code that matches intent.
再看同一个请求，但这次带上规约：'用户可以上传 JPEG 或 PNG 格式的照片，大小不超过 10MB；照片存储在 S3 中，key 以用户 ID 作为前缀；只有上传者可以删除自己的照片；上传时照片会被调整到最长边 1024 像素。'这下 AI 就有足够信息生成符合意图的代码了。
Key Insight — The Core Principle: In spec-driven development, code is the implementation detail of the specification—not the other way around. The spec declares intent; the code realizes it.
核心洞见 —— 核心原则： 在规约驱动开发中，代码是规约的"实现细节"，而不是反过来。规约宣告意图，代码负责兑现。

A. Spec-First: Guided Initial Development
A. 规约先行（Spec-First）：引导初始开发
Definition — Spec-First: In spec-first development, a specification is written before coding to guide the initial implementation. Once code exists, the spec may or may not be maintained—the primary value is in the initial clarity it provides.
定义 —— 规约先行： 在规约先行模式中，规约是在写代码之前就写好的，用来引导最初的实现。一旦代码存在，这份规约可能会被继续维护，也可能不会——它的核心价值在于"起步时提供清晰度"。
Spec-first represents the entry point to SDD. Before writing code, the developer or team articulates what the code should do, typically as a user story with acceptance criteria, a BDD scenario, or a detailed requirements document. The spec guides implementation, but once the code is written and tests pass, the spec may be discarded or allowed to drift.
规约先行是进入 SDD 的入口层。在动手写代码之前，开发者或团队先明确代码应该做什么，通常以"带验收标准的用户故事"、"一段 BDD 场景"或"一份详细的需求文档"的形式写出来。这份规约用于引导实现，但一旦代码写完、测试通过，它就可能被丢到一边，或者被允许逐渐漂移。
The defining characteristic of spec-first development is that the specification is written before implementation begins, ensuring that developers have a clear target before they start coding. However, the code becomes the primary artifact post-implementation, and the spec may become outdated as the code evolves through subsequent iterations. This approach carries a lower maintenance burden than stronger specification disciplines, making it practical for teams that cannot commit to ongoing spec maintenance.
规约先行最显著的特征，是在实现开始之前就已经写好了规约——这保证开发者在动笔之前已经有了明确的靶子。但实现完成之后，代码会成为主导产物，而规约在后续迭代中可能会过时。这种做法比更强的规约纪律维护成本更低，对于那些无法持续投入维护规约的团队来说是个务实的选择。
Spec-first works particularly well for initial feature development when working with AI coding assistants. The upfront spec prevents the AI from guessing at requirements, dramatically improving the quality of generated code. It is also valuable for prototypes and one-off features where the cost of maintaining a spec alongside code indefinitely is not justified. However, spec-first does not protect against drift over time—if the codebase will be maintained long-term, teams should consider spec-anchored approaches.
规约先行尤其适合"在 AI 编程助手协助下做新功能的初次开发"：前置的规约能阻止 AI 去"猜"需求，显著提升生成代码的质量。它对原型和一次性功能也有价值——这类场景下"长期维护一份规约"的成本并不划算。不过，规约先行并不能防止"随时间漂移"：如果这段代码将会长期维护，团队应该考虑改用规约锚定的做法。
B. Spec-Anchored: Living Documentation
B. 规约锚定（Spec-Anchored）：活文档
Definition — Spec-Anchored: In spec-anchored development, the specification is maintained alongside the code throughout the system's lifecycle. Changes to behavior require updating both the spec and the code, keeping them synchronized.
定义 —— 规约锚定： 在规约锚定模式中，规约在系统的整个生命周期中都与代码并行维护。任何行为上的变更都要求同时更新规约和代码，保持二者同步。
Spec-anchored development treats the spec as a living document that evolves with the codebase. When a feature changes, the spec is updated first or simultaneously with the code. Automated checks—typically in the form of tests derived from the spec—ensure that spec and code remain aligned. If they drift, tests fail, providing immediate feedback that the system's documentation no longer reflects its behavior.
规约锚定把规约当作一份"活文档"，与代码库一同演化。当一个功能发生变化时，规约要么先更新，要么与代码同步更新。自动化检查——通常以"从规约派生出的测试"的形式——保证规约和代码始终对齐。一旦二者漂移，测试就会失败，立即反馈给团队："系统的文档已经不能再反映它的实际行为了。"
In this approach, the specification and code evolve together as equal partners. Tests enforce the alignment between them, with BDD scenarios commonly serving as automated tests that run on every commit. The spec serves as always-up-to-date documentation that developers and stakeholders can trust. However, maintaining this alignment requires discipline and tooling support—teams must commit to updating specs whenever behavior changes.
在这种模式下，规约和代码作为平等的搭档一同演化。测试强制它们对齐，其中 BDD 场景通常作为每次提交都会跑的自动化测试。规约扮演"永远保持最新"的文档，开发者和利益相关方都可以信任它。但要维持这种对齐，需要纪律和工具支撑——只要行为有变，团队就必须同步更新规约。
Spec-anchored is the sweet spot for most production systems. It provides the benefits of clear documentation and verifiable requirements without demanding that code be fully generated from specifications. BDD frameworks like Cucumber exemplify this approach, enabling teams to write human-readable scenarios that execute as automated tests. For API development, OpenAPI specifications paired with contract testing tools like Specmatic achieve the same alignment between spec and implementation.
规约锚定是绝大多数生产系统的最佳落点。它既提供了清晰文档和可验证需求的好处，又不强求代码必须完全由规约生成。Cucumber 这样的 BDD 框架就是这种做法的典型，让团队能写出人类可读的场景，同时作为自动化测试执行。在 API 开发中，OpenAPI 规约搭配 Specmatic 之类的契约测试工具，也能实现"规约与实现对齐"的效果。
C. Spec-as-Source: Humans Edit Specs, Machines Generate Code
C. 规约即源（Spec-as-Source）：人只改规约，机器生成代码
Definition — Spec-as-Source: In spec-as-source development, the specification is the only artifact humans edit directly. Code is entirely generated from the spec and should never be manually modified. Any change to behavior means changing the spec and regenerating.
定义 —— 规约即源： 在规约即源模式中，规约是人类唯一直接编辑的产物。代码完全由规约生成，不应被手工修改。任何行为变更都意味着先改规约，再重新生成代码。
Spec-as-source represents the most radical form of SDD. The specification becomes, in effect, the source code—just expressed at a higher level of abstraction. Developers think in terms of requirements and behavior; machines translate that into executable code. If you want to change functionality, you change the spec and regenerate—you never edit the generated code directly.
规约即源是 SDD 最激进的一种形态。此时规约实际上变成了"源代码"——只不过表达在更高的抽象层次上。开发者在需求和行为的层面思考，由机器把这些东西翻译成可执行代码。如果你想改功能，就改规约并重新生成——你永远不直接编辑生成出来的代码。
This approach, drawing on Design by Contract principles [24], fundamentally inverts the traditional relationship between specs and code: the specification is the primary artifact, and code is entirely derived from it. Manual code editing is either prohibited or confined to well-defined extension points. This requires mature, trusted generation tooling—developers must have confidence that generated code correctly implements the spec. In return, drift is eliminated by design: since code is regenerated rather than manually edited, spec and code are always aligned by construction.
这种做法借鉴了"契约式设计（Design by Contract）"的思想 [24]，从根本上颠倒了规约与代码的传统关系：规约是主导产物，代码完全由它派生。手工改代码要么被禁止，要么只被允许在界定清晰的扩展点上进行。这要求生成工具链足够成熟、足够可信——开发者必须相信生成出来的代码确实正确地实现了规约。作为交换，"漂移"被从设计上消除了：既然代码是重新生成而非手工改出来的，规约和代码就从构造上始终保持对齐。
Spec-as-source is already standard practice in domains with well-defined code generation, such as generating API server stubs from OpenAPI specifications, or producing certified embedded code from Simulink models. In the automotive industry, engineers routinely build control algorithms in Simulink, verify behavior at the model level through simulation, and generate certified C code that nobody hand-edits. Emerging AI tools like Tessl aim to extend this approach to general software development, representing a future where specifications truly become "the new source code." However, adopting spec-as-source requires high trust in generation quality and is currently practical only in domains where that trust has been established.
在一些代码生成已经非常成熟的领域，规约即源已是标准做法，例如根据 OpenAPI 规约生成 API 服务端桩代码，或者从 Simulink 模型产出经过认证的嵌入式代码。在汽车行业，工程师常常用 Simulink 搭建控制算法，在模型层通过仿真验证其行为，然后生成认证过的 C 代码——没人会去手改。像 Tessl 这样的新兴 AI 工具正在尝试把这种做法推广到通用软件开发，描绘出"规约真正变成新的源代码"的未来。但要采用规约即源，前提是对生成质量有高度信任，目前只有在这种信任已经建立的领域里才真正可行。

A. Phase 1: Specify
A. 阶段一：规约（Specify）
The specify phase answers a fundamental question: What should the software do? The output is a functional specification describing behavior, requirements, and acceptance criteria—crucially, without prescribing implementation details. This separation of "what" from "how" is essential to SDD's power.
规约阶段要回答一个根本问题：软件应该做什么？它的产出是一份功能规约，描述行为、需求与验收标准——关键是，这里不规定"怎么实现"。把"做什么"从"怎么做"里分离出来，是 SDD 力量的根源。
During this phase, teams articulate user-facing behavior through user stories, scenarios, and acceptance criteria. They define what success looks like using Given/When/Then format or input-output examples. Business rules and constraints are captured explicitly, and edge cases and error conditions are identified upfront rather than discovered during implementation.
在这个阶段，团队通过用户故事、场景和验收标准来表达面向用户的行为；用 Given/When/Then（当/若/则）格式或输入-输出示例来定义"成功是什么样"。业务规则和约束被显式写下来，边界情况和错误条件也在前期就被识别，而不是在实现过程中才被发现。
The quality of specifications directly determines the quality of everything that follows. Good specs share several characteristics: they are behavior-focused, describing what happens rather than how; they are testable, with each requirement being verifiable; they are unambiguous, meaning different readers reach the same interpretation; and they are complete enough to cover essential cases without over-specifying. Effective specs emphasize clarity, modularity, and self-checks that help guide AI agents during implementation.
规约的质量直接决定了后面所有环节的质量。好的规约通常具备这几个特征：聚焦行为，描述会发生什么而不是怎么发生；可测试，每条需求都可被验证；无歧义，不同读者读到的是同一个意思；足够完整，能覆盖关键情形，但不过度规定。有效的规约会强调清晰、模块化和自检点，帮助 AI 智能体在实现阶段按图施工。
Practitioner's Tip: Write specs at the level of detail needed to remove ambiguity. If an AI or developer could interpret a requirement in multiple ways, add clarification. If there's only one reasonable interpretation, don't over-specify—excessive detail constrains implementation unnecessarily.
从业者建议： 把规约写到"足以消除歧义"的详细程度。如果一条需求 AI 或开发者可能有多种解读，就加一句澄清；如果只有一种合理解读，就别再进一步细化——过度细节会不必要地束缚实现。
B. Phase 2: Plan
B. 阶段二：规划（Plan）
The plan phase answers a different question: How should we build it? Given the functional spec, this phase produces a technical plan covering architecture, data models, interfaces, and technology choices. Where the spec declares intent, the plan declares constraints on implementation.
规划阶段回答的是另一个问题：我们应该怎么把它建出来？在功能规约的基础上，本阶段产出一份技术规划，涵盖架构、数据模型、接口与技术选型。规约宣告意图，而规划则宣告对实现的"约束"。
Planning involves selecting technologies and frameworks appropriate to the problem, defining component architecture and boundaries, designing data models and schemas, specifying interfaces including APIs, messages, and contracts, and identifying non-functional requirements around performance, security, and scalability.
规划涉及：为问题选择合适的技术与框架，定义组件架构与边界，设计数据模型与模式，规定接口（包括 API、消息和契约），以及识别围绕性能、安全和可扩展性的非功能性需求。
The plan phase bridges the "what" and the "how." It encodes constraints that the implementation must respect—for example, "use PostgreSQL for persistence" or "all API endpoints require authentication." When using AI coding assistants, the plan provides crucial context: the AI learns not just what to build but how the system is structured and what conventions it should follow. Without this context, even a perfect functional spec may yield code that contradicts organizational standards or architectural decisions.
规划阶段是"做什么"与"怎么做"之间的桥梁。它把实现必须遵守的约束编码下来——例如"用 PostgreSQL 做持久化"或者"所有 API 端点都要认证"。在使用 AI 编程助手时，这份规划提供了关键上下文：AI 学到的不仅是"要建什么"，还有系统是怎么组织的、应该遵循哪些约定。没有这份上下文，即便一份完美的功能规约，也可能产出违背组织标准或架构决策的代码。
C. Phase 3: Implement
C. 阶段三：实现（Implement）
The implement phase produces working code that realizes the spec according to the plan. In traditional development, this is where most effort concentrates. In SDD, particularly with AI assistance, this phase may be substantially automated—but it still requires human oversight.
实现阶段产出真正工作的代码，按照规划把规约落地。在传统开发里，大多数精力都集中在这里。在 SDD 里，尤其是在有 AI 辅助的情况下，这个阶段可以被大幅自动化——但仍然需要人的监督。
Implementation begins by breaking the plan into discrete, reviewable tasks. Each task is then implemented, whether by human developers, AI assistants, or a hybrid approach. Code is reviewed against both spec and plan to verify alignment. Unit tests are written to encode spec requirements as executable assertions.
实现通常先把规划拆成一个个离散、可评审的任务；每个任务再被实现，可能由人类开发者、AI 助手，或二者混合完成。代码要对着规约和规划进行评审以验证对齐。单元测试则把规约里的需求编码成可执行的断言。
A key SDD principle is working in small, validated increments. Rather than implementing the entire spec at once, teams break work into tasks where each delivers a testable piece of functionality. This enables frequent checkpoints where humans verify alignment, catching drift early before it compounds. Specifications act as "super-prompts" that break down complex problems into modular components aligned with agents' context windows, enabling AI systems to handle complexity that would overwhelm single-shot prompts.
SDD 的一个关键原则是"以小步、带验证的增量推进"。与其一口气把整份规约全部实现掉，团队更应该把工作切成任务，让每个任务都交付一块可测试的功能。这能制造出频繁的检查点，让人去核对对齐情况，早早发现漂移，避免它越积越多。规约可以被当作"超级提示（super-prompt）"，把复杂问题拆成与智能体上下文窗口匹配的模块化组件，让 AI 系统能处理单次提示根本应付不了的复杂度。
D. Phase 4: Validate
D. 阶段四：验证（Validate）
The validate phase answers the crucial closing question: Does the code actually meet the spec? Validation closes the loop, ensuring that what was specified is what was built. This phase combines automated verification with human judgment.
验证阶段回答那个关键的收尾问题：代码真的满足规约吗？验证把回路闭合，确保"规定出来的"就是"建出来的"。这一阶段把自动化验证与人的判断结合起来。
Validation encompasses running automated tests at unit, integration, and acceptance levels, executing BDD scenarios against the implementation, reviewing for adherence to non-functional requirements, and conducting stakeholder acceptance testing where appropriate.
验证涵盖：在单元、集成和验收层级上运行自动化测试，针对实现执行 BDD 场景，审查对非功能性需求的符合度，以及在合适的时候进行利益相关方验收测试。
If validation reveals gaps—the code doesn't meet the spec—the team faces a decision: fix the code or revise the spec. If the original spec was wrong or incomplete, updating it is the right choice. If the code simply doesn't meet a valid spec, fixing the code is required. Either way, the spec remains the authority. This discipline ensures that specifications stay trustworthy—teams can rely on them because violations are detected and addressed, not ignored.
如果验证发现了缺口——代码不满足规约——团队就要做一个选择：修代码，还是改规约？如果原规约本身是错的或不完整，那么更新规约才是正确做法；如果只是代码没满足一份合理的规约，那就必须修代码。无论哪种情况，规约始终是权威。正是这种纪律让规约保持可信：团队之所以敢依赖规约，是因为违反之处会被发现和处理，而不是被忽视。
IV. How SDD Boosts AI Coding Agents
四、SDD 如何增强 AI 编程智能体
Large language models like GPT-4 or Claude, when used as coding agents, benefit immensely from SDD by receiving optimized, context-rich inputs. Specifications act as super-prompts that break down complex problems into modular components aligned with agents' context windows. AI agents can generate code from specs while self-verifying against checklists for requirements adherence.
像 GPT-4 或 Claude 这样的大型语言模型，作为编程智能体使用时，通过接收优化过、语境丰富的输入，从 SDD 中获益极大。规约扮演"超级提示"的角色，把复杂问题拆成与智能体上下文窗口匹配的模块化组件。AI 智能体可以从规约出发生成代码，同时对照清单自检是否满足需求。
Empirical studies [5][6], though nascent, suggest that human-refined specs significantly improve LLM-generated code quality, with controlled studies showing error reductions of up to 50%. This boosting effect is particularly evident in scalable scenarios: specifications enable parallel agent execution on non-overlapping tasks, with orchestration for dependencies. Teams can partition work at the spec level, allowing multiple AI agents to implement different components simultaneously without interference.
现有的实证研究 [5][6] 虽仍处早期，但提示：人工打磨过的规约能显著提升大模型生成代码的质量——有对照实验显示错误率最多下降 50%。这种增强效应在可规模化的场景中尤为明显：规约使"多个智能体并行处理彼此不重叠的任务"成为可能，并配合对依赖关系的编排。团队可以在"规约层"切分工作，让多个 AI 智能体同时实现不同组件而互不干扰。
Challenges remain, including LLM non-determinism—even structured specs can lead to varying outputs. Techniques like property-based testing (PBT) address this by automatically verifying that invariants from specs are satisfied regardless of implementation variation. In embedded systems and other safety-critical domains, SDD combines LLM generation with formal verification to ensure compliance with standards like ISO 26262 [25]. Overall, SDD transforms AI agents from reactive tools into proactive collaborators, enhancing efficiency particularly in brownfield projects where legacy constraints are encoded in specifications.
挑战依然存在，其中之一是大模型的非确定性——即便给了结构化规约，输出也可能每次不同。像"基于属性的测试（PBT）"这类技术可以缓解这一点，它会自动验证"来自规约的不变式"是否被满足，不论实现上有多少变化。在嵌入式系统等安全关键领域，SDD 把大模型生成与形式化验证结合起来，确保符合 ISO 26262 [25] 这类标准。整体而言，SDD 把 AI 智能体从"被动工具"转变成"主动协作者"，尤其在"棕地项目（brownfield）"里效率提升明显——在那里，历史遗留的约束会被写进规约。
An emerging approach involves "self-spec" methods where LLMs author their own specifications before generating code. The agent first produces a spec from a high-level prompt, which is then reviewed and refined by humans before the same or another agent implements against it. This creates an explicit separation between planning and execution, catching requirement misunderstandings before code is written.
一种新兴做法是"自写规约（self-spec）"：让大模型在生成代码之前，先自己起草一份规约。智能体先根据高层提示产出规约，人类对其进行评审和打磨，然后让同一个或另一个智能体按这份规约去实现。这样就在"规划"和"执行"之间做出了显式区分，在写代码之前就抓住了对需求的误解。

A. Behavior-Driven Development (BDD) Frameworks
A. 行为驱动开发（BDD）框架
BDD frameworks allow teams to write specifications in near-natural language that can be executed as tests. The canonical format is Gherkin [16], which uses structured scenarios with Given/When/Then clauses:
BDD 框架允许团队用接近自然语言的方式写规约，而这些规约可以作为测试执行。它的规范化格式是 Gherkin [16]，使用带 Given/When/Then 子句的结构化场景：
Feature: Shopping Cart
  Scenario: Adding item to empty cart
    Given the cart is empty
    When I add item "Widget" to the cart
    Then the cart should contain 1 item
    And the item should be "Widget"

  Scenario: Adding item to empty cart
    Given the cart is empty
    When I add item "Widget" to the cart
    Then the cart should contain 1 item
    And the item should be "Widget"
These scenarios serve dual purposes: documentation that stakeholders can read and automated tests that verify code. Tools like Cucumber [7] (Ruby, Java, JavaScript), SpecFlow [21] (.NET), and Behave [22] (Python) execute these scenarios against the application, bridging the gap between business requirements and technical implementation.
这些场景承担双重角色：既是利益相关方可读的文档，也是验证代码的自动化测试。Cucumber [7]（Ruby、Java、JavaScript）、SpecFlow [21]（.NET）、Behave [22]（Python）等工具会让这些场景在应用上运行，从而在业务需求和技术实现之间架起桥梁。
Practitioner's Tip: BDD scenarios are specifications, not just tests. Write them before implementation, involve stakeholders in their creation, and treat them as the authoritative description of feature behavior. When scenarios pass, you have confidence that the system meets its documented requirements.
从业者建议： BDD 场景是规约，不只是测试。在实现之前就写好它们，让利益相关方参与编写，并把它们当作对功能行为的权威描述。当场景全部通过时，你就有信心认定：系统满足了它被记录下来的需求。
B. API Specification Tools
B. API 规约工具
In API development, spec-driven approaches have been standard practice under the names "design-first" or "API-first" [23] for years. OpenAPI [8] (formerly Swagger) enables teams to define REST APIs with complete endpoint specifications, request/response schemas, and examples, then generate server stubs, client SDKs, and documentation from the spec. GraphQL SDL [17] allows teams to define types, queries, and mutations in a schema that becomes the contract between frontend and backend, enabling parallel development. For event-driven architectures, AsyncAPI [27] provides similar specification capabilities. Protocol Buffers [18] and gRPC [19] enable definition of service interfaces and message types with automatic generation of strongly-typed client and server code.
在 API 开发领域，规约驱动的做法多年来早已以"设计先行（design-first）"或"API 先行（API-first）"[23] 的名字成为标准实践。OpenAPI [8]（以前叫 Swagger）让团队可以完整定义 REST API 的端点规约、请求/响应模式和示例，然后从规约生成服务端桩代码、客户端 SDK 和文档。GraphQL SDL [17] 让团队用一份 schema 定义类型、查询和变更，这份 schema 就是前后端之间的契约，可以支持并行开发。对于事件驱动架构，AsyncAPI [27] 提供了类似的规约能力。Protocol Buffers [18] 和 gRPC [19] 则允许定义服务接口和消息类型，并自动生成强类型的客户端与服务端代码。
The benefit of API specification tools is clear: once the API spec is agreed upon, frontend and backend teams can work in parallel with confidence. The spec is the contract; any implementation that matches the spec is valid by definition. Contract testing tools like Pact [20] and Specmatic [9] automate verification that implementations actually match their specs.
API 规约工具的好处很直接：一旦 API 规约达成一致，前后端团队就可以放心地并行工作。规约就是契约；任何符合规约的实现按定义都算有效。像 Pact [20] 和 Specmatic [9] 这样的契约测试工具可以自动验证"实现确实与其规约匹配"。
C. AI-Assisted SDD Tools
C. AI 辅助的 SDD 工具
Emerging tools structure AI coding workflows explicitly around specifications, recognizing that multi-step prompting with explicit artifacts yields better results than single-shot "just code this" prompts.
一批新工具围绕"规约"明确地重塑了 AI 编程工作流，因为人们意识到：带显式产物的多步提示，比单步"就把这段写了"的提示，效果好得多。
GitHub Spec Kit [1] is an open-source toolkit providing commands for spec-driven AI development. The workflow follows four explicit phases: /specify generates a detailed spec from a prompt, /plan creates technical architecture, /tasks breaks the plan into implementation tasks, and finally implementation generates code task by task. At each phase, humans review and refine before proceeding, maintaining alignment between intent and implementation.
GitHub Spec Kit [1] 是一个开源工具包，为"规约驱动的 AI 开发"提供命令。其工作流分四个显式阶段：/specify 根据提示生成详细规约，/plan 制定技术架构，/tasks 把规划拆成实现任务，最后在实现阶段按任务逐个生成代码。每个阶段人类都会先评审、打磨再继续，保持意图和实现之间的对齐。
Amazon Kiro [13] guides users through requirements, design, and task creation stages before any code generation begins. Kiro emphasizes structured requirements capture and iterative refinement, ensuring AI has clear context before attempting implementation. The explicit staging prevents the AI from guessing at requirements that were never specified.
Amazon Kiro [13] 引导用户依次经过需求、设计和任务创建阶段，然后才开始生成代码。Kiro 强调结构化的需求捕获与迭代打磨，确保 AI 在尝试实现之前已经有了清晰上下文。这种显式分段能阻止 AI 去"猜"那些从未被指定的需求。
Tessl [14] takes the most radical approach: spec-as-source where the specification is the maintained artifact and code is regenerated from it. Tessl represents the emerging vision of "specs as the new source code," where developers never edit generated code—they edit specs and regenerate.
Tessl [14] 走的是最激进的路线：规约即源——规约是被维护的产物，代码由它重新生成。Tessl 代表了"规约作为新的源代码"这一正在出现的愿景：开发者永远不编辑生成的代码，他们编辑规约并重新生成。
These tools share a common insight: separating planning from implementation allows agents to focus on execution within defined boundaries, reducing the non-determinism that plagues loosely-prompted AI coding.
这些工具都体现出一个共同洞见：把规划与实现分开，能让智能体在已界定的边界内专注执行，从而减少那种折磨着"松散提示式 AI 编程"的非确定性。

A. Case Study 1: API-First Microservices
A. 案例研究一：API 先行的微服务
Domain: Financial services microservices
Pattern: Spec-anchored with OpenAPI
Outcome: 75% reduction in integration cycle time
领域: 金融服务业的微服务
模式: 用 OpenAPI 做规约锚定
结果: API 变更的集成周期时间缩短 75%
A financial services company struggled with what they called "integration hell"—microservices frequently failed when deployed together because teams made incompatible assumptions about API contracts. Each team implemented their service in isolation, and incompatibilities only surfaced during integration testing, requiring expensive rework.
一家金融服务公司深受他们自己称为"集成地狱"的问题困扰——多个微服务在一起部署时频繁失败，因为各团队对 API 契约做出的假设互不兼容。每个团队都在孤岛里实现自己的服务，不兼容的问题直到集成测试才浮出水面，代价高昂的返工随之而来。
The company mandated API-first development as their solution. Before implementing any service, teams wrote OpenAPI specifications defining endpoints, request/response schemas, and error conditions. Consumer teams reviewed these specs and provided feedback before any coding began. This front-loaded the integration discussions that previously happened too late.
这家公司的对策是强制推行 API 先行开发。在实现任何服务之前，团队先写 OpenAPI 规约，把端点、请求/响应模式和错误条件定义清楚。消费方团队在任何代码动笔之前就先评审这份规约、给出反馈。这样一来，以前那些拖得太晚才发生的集成讨论，被提前到了项目前期。
They used Specmatic [9] to generate mock servers from specs, allowing frontend development to proceed in parallel with backend work. More critically, Specmatic validated that implemented services matched their specs in CI. Any deviation caused the build to fail, preventing drift from accumulating.
他们使用 Specmatic [9] 从规约生成 mock 服务器，让前端开发可以和后端工作并行推进。更关键的是，Specmatic 在 CI 里校验实现的服务是否真的匹配规约。一旦出现偏差，构建就会失败，从而防止漂移累积。
Integration failures dropped dramatically after adoption. Teams reported a 75% reduction in cycle time for API changes because incompatibilities were caught at the spec review stage rather than in production. The specs became the contract that all parties trusted, eliminating the ambiguity that had caused so much rework.
采用之后，集成失败率大幅下降。团队报告说 API 变更的周期时间缩短了 75%——因为不兼容的问题在"规约评审"阶段就被抓住了，而不是拖到生产环境。规约成了各方都信任的契约，消除了以往造成大量返工的歧义。
B. Case Study 2: BDD for Enterprise Features
B. 案例研究二：企业功能中的 BDD
Domain: Enterprise project management software
Pattern: Spec-anchored with Cucumber
Outcome: Stakeholder-verifiable requirements; reduced requirement ambiguity
领域: 企业级项目管理软件
模式: 用 Cucumber 做规约锚定
结果: 需求可被利益相关方验证；需求歧义减少
An enterprise software team found that developers and product managers frequently disagreed on what "done" meant for features. Developers would implement something they believed met requirements, QA would find it didn't match product expectations, and arguments ensued about whose interpretation was correct. The lack of a shared, authoritative definition of expected behavior caused friction and rework.
一个企业软件团队发现，开发者和产品经理经常在"功能做完是什么意思"上意见相左。开发者会按他们自己认为满足需求的方式去实现，随后 QA 发现这与产品的期望并不一致，于是大家就开始争论"到底谁的理解才对"。对预期行为缺乏一个共享且权威的定义，造成了摩擦与返工。
The team adopted Cucumber for all user-facing features as their solution. Product managers wrote Gherkin scenarios describing expected behavior in plain language. Developers implemented step definitions to automate these scenarios as executable tests. A feature was only considered "done" when all its scenarios passed, providing an objective, verifiable definition of completion.
他们的对策是：在所有面向用户的功能上采用 Cucumber。产品经理用 Gherkin 以平实语言写场景来描述预期行为；开发者则实现步骤定义，把这些场景自动化为可执行测试。只有当一个功能的全部场景都通过时，它才算"做完"，从而为"完成"提供了一个客观、可验证的定义。
The Gherkin scenarios became a shared language that both business and technical stakeholders could read and validate. Product managers could verify that scenarios captured their intent. When disputes arose, the scenario was the authority—if the scenario was wrong, it was updated with explicit agreement from stakeholders; if the code was wrong, developers fixed it. This eliminated the ambiguity that had caused so much rework and conflict.
这些 Gherkin 场景成了一种业务和技术双方都能读、都能验证的共享语言。产品经理可以核实场景是否抓住了他们的意图。出现分歧时，场景本身就是权威——如果场景写错了，就在利益相关方明确同意下更新它；如果代码写错了，开发者就去修。这就消除了以往引起大量返工和冲突的歧义。
C. Case Study 3: Model-Based Embedded Development
C. 案例研究三：基于模型的嵌入式开发
Domain: Automotive engine control
Pattern: Spec-as-source with Simulink
Outcome: Verified control logic; certified code generation
领域: 汽车发动机控制
模式: 基于 Simulink 的规约即源
结果: 控制逻辑得到验证；代码生成通过认证
An automotive supplier needed to develop engine control software that met ISO 26262 safety certification requirements. Manual coding was error-prone, and certification required tracing every line of code to specific requirements—a labor-intensive process when code was hand-written.
一家汽车供应商需要开发满足 ISO 26262 安全认证要求的发动机控制软件。手工编码容易出错，而认证又要求把每一行代码追溯到具体的需求——在代码是手写的情况下，这是一项极费人力的工作。
The team used MathWorks Simulink [10] to model control algorithms as block diagrams with state machines. The model was the specification: engineers simulated and verified behavior at the model level, catching algorithmic errors before any code existed. Once the model was verified through simulation, code was auto-generated from the model using a certified code generator.
他们的做法是使用 MathWorks Simulink [10]，把控制算法建模为带状态机的框图。这份模型就是规约：工程师在模型层进行仿真和验证，能在任何代码存在之前就捕获算法错误。模型通过仿真验证后，使用经过认证的代码生成器从模型自动生成代码。
The model-to-code generation was itself certified, meaning the generated C code was guaranteed to behave as the model specified. Engineers never edited generated code; if control logic needed changing, they changed the model and regenerated. This ensured the verified model and deployed code remained perfectly aligned by construction.
"从模型到代码"这一步本身也是经过认证的，这意味着生成出的 C 代码在行为上被保证与模型所规定的一致。工程师从不编辑生成的代码；如果控制逻辑需要修改，他们去改模型，再重新生成。这从构造上保证了"通过验证的模型"与"部署的代码"始终完美对齐。
This approach embodies spec-as-source at its most rigorous: the specification (Simulink model) is the only artifact humans modify, and the implementation (C code) is entirely generated. SDD in embedded systems combines LLM generation with formal verification to ensure safety-critical compliance, demonstrating how specs ensure precision in domains like automotive and aerospace where errors can be catastrophic.
这种做法体现了规约即源最严格的形态：规约（Simulink 模型）是人类唯一会修改的产物，而实现（C 代码）完全由其生成。嵌入式系统中的 SDD 把大模型生成与形式化验证结合起来，以确保安全关键场景的合规性；这也说明了在汽车、航空航天等"一旦出错就可能是灾难性"的领域里，规约是如何保证精度的。
VII. The Redefinition of Developer Work
七、开发者工作的重新定义
SDD fundamentally reshapes what it means to be a software developer. Work is being redefined as developers shift from manual coding to orchestrating specifications, reviewing AI outputs, and focusing on high-level design. This transition potentially increases efficiency but introduces new challenges around spec maintenance, tool mastery, and the judgment required to know when AI outputs are correct.
SDD 从根本上重塑了"当一个软件开发者"意味着什么。开发者的工作正在被重新定义：从手工写代码，转向编排规约、评审 AI 的输出、聚焦高层设计。这一转变有可能提升效率，但也带来了新的挑战——关于规约维护、工具掌握，以及判断 AI 输出是否正确所需要的判断力。
In greenfield projects, developers become architects who design systems through specifications rather than code. They focus on requirements elicitation, constraint definition, and acceptance criteria—the "what" rather than the "how." AI agents handle the translation from spec to implementation, but humans remain responsible for ensuring specs capture actual requirements.
在"绿地项目（greenfield）"里，开发者成了通过规约而非代码来设计系统的架构师。他们关注需求提取、约束定义和验收标准——即"做什么"，而不是"怎么做"。从规约到实现的翻译交给 AI 智能体，但"规约是否真的抓住了实际需求"仍然是人的责任。
In brownfield projects and legacy systems, SDD enables a different kind of work: encoding existing behavior as specifications before making changes. By extracting specs from legacy code, teams can verify that modernization efforts preserve required functionality while eliminating undocumented behaviors. The spec becomes the bridge between old and new implementations.
在棕地项目和遗留系统里，SDD 带来的是另一种工作方式：在改动之前，先把现有行为编码成规约。通过从遗留代码中抽取规约，团队可以验证现代化改造是否保留了所需功能，同时清理掉那些未被记录的行为。规约成了新旧实现之间的桥梁。
Use-cases span the development spectrum: greenfield projects where specs guide initial development, feature additions to legacy systems where specs document existing behavior before modification, and embedded software where specs ensure precision in safety-critical domains. In each case, the developer's role shifts from code producer to spec author and AI orchestrator.
用例覆盖了整个开发谱系：在绿地项目中，规约引导最初的开发；在给遗留系统加功能时，规约先记录现有行为再进行修改；在嵌入式软件中，规约保证了安全关键领域的精度。在每种情况下，开发者的角色都从"代码生产者"变成了"规约作者"和"AI 编排者"。
VIII. When to Use SDD
八、什么时候该用 SDD

SDD adds clear value when using AI coding assistants, as specifications dramatically improve output quality by removing the ambiguity that forces AI to guess. Complex requirements benefit from SDD because stakeholders can validate that the system meets their needs before code is written. Systems with multiple maintainers benefit because specs serve as documentation that survives team turnover. Integration-heavy systems gain from API specs that enable parallel development and prevent integration failures. Regulated domains often mandate traceability from requirements to implementation, which SDD provides naturally. Legacy modernization efforts benefit because extracting a spec from existing behavior enables clean reimplementation with confidence.
SDD 在使用 AI 编程助手时，带来的价值非常明确：规约能消除那种逼得 AI 去"猜"的歧义，从而显著提高输出质量。复杂需求也能从 SDD 中获益，因为利益相关方可以在代码写下去之前就验证系统是否满足他们的需要。有多个维护者的系统能从中受益，因为规约承担着"能熬过团队人员更替"的文档角色。集成繁重的系统可以借助 API 规约实现并行开发、避免集成失败。受监管的领域通常要求从需求到实现可追溯，而这正是 SDD 天然提供的。遗留系统的现代化改造也能获益，因为从现有行为中抽取规约，可以让"干净地重新实现"变得有信心。
However, SDD may be overkill in certain situations. Throwaway prototypes don't justify spec investment that will be discarded. Solo, short-lived projects may find the overhead exceeds benefits when there's only one developer and no long-term maintenance. Exploratory coding suffers from premature specification that constrains learning when you don't yet know what you're building. Simple CRUD applications with obvious requirements need minimal spec—if requirements are unlikely to be misinterpreted, elaborate specifications add cost without value.
不过，在某些情况下 SDD 就显得过度了。一次性用完就扔的原型，不值得做那种"注定要丢掉"的规约投入。独立开发、寿命短的项目里，当只有一个开发者且没有长期维护时，开销往往超过收益。探索式编程如果过早做规约，反而会束缚学习——你还不知道自己到底在造什么呢。那些需求显而易见的简单 CRUD 应用只需要极少的规约——如果需求基本不会被误解，精心打磨的规约就是增加成本却没有价值。
Key Insight — The Golden Rule: Use the minimum level of specification rigor that removes ambiguity for your context. Spec-first for AI-assisted initial development; spec-anchored for long-lived production systems; spec-as-source only when generation tooling is mature and trusted.
核心洞见 —— 黄金法则： 采用"恰好能消除歧义"的最低规约严格度。AI 辅助的初次开发用规约先行；长期运行的生产系统用规约锚定；只有在生成工具链足够成熟、足够可信时，才去用规约即源。
IX. Common Pitfalls
九、常见陷阱
Teams adopting SDD often encounter predictable challenges that can undermine the approach's benefits if not addressed.
团队在采用 SDD 的过程中，经常会遇到一些可预见的挑战；如果不加以应对，它们会把 SDD 的好处抵消掉。
Over-specification occurs when teams write specs that are too detailed, essentially becoming pseudo-code. This defeats the purpose of SDD, which is to separate "what" from "how." If your spec reads like code, you've gone too far—you've constrained implementation unnecessarily and lost the abstraction benefit that makes specs valuable.
过度规约出现在团队把规约写得太细、实际上变成了伪代码的时候。这违背了 SDD 的初衷——把"做什么"和"怎么做"分开。如果你的规约读起来像代码，那就走过头了：你不必要地束缚了实现，丢掉了那种让规约之所以有价值的抽象收益。
Specification rot affects spec-anchored approaches when teams fail to update specs as code changes. The spec drifts from reality, losing its value as documentation and eroding trust. The solution is automated enforcement through tests that fail when spec and code diverge, making drift visible and painful rather than silent and accumulating.
规约腐烂是规约锚定模式下的典型问题，出现在团队没能随着代码变化去更新规约时。规约会逐渐偏离现实，失去作为文档的价值，信任也随之被侵蚀。解法是通过测试来自动化地强制执行——当规约和代码出现偏离时让测试失败，把漂移变成"显眼且难受"的，而不是"悄悄积累"的。
Specification as bureaucracy emerges when specs become forms to fill out rather than tools for clarity. If the specification process adds overhead without improving understanding or quality, teams will game the system or abandon it. Specs should be the minimum needed to remove ambiguity, not comprehensive documentation exercises.
规约作为官僚程序出现在规约变成"要填的表格"而不是"提升清晰度的工具"的时候。如果规约流程只增加负担，却不改进理解或质量，团队要么会敷衍了事，要么会索性放弃它。规约应该是"刚好消除歧义所需要的最少量"，而不是一场全面的文档练习。
Tooling complexity can overwhelm teams, particularly with AI-assisted tools that generate elaborate artifacts. Teams may drown in generated plans, task lists, and intermediate documents. The solution is to start simple and add tooling complexity only when it demonstrably helps—avoid cargo-culting elaborate workflows that add process without value.
工具复杂度也可能压垮团队，尤其是那些会生成一大堆产物的 AI 辅助工具。团队可能被生成出来的规划、任务列表和中间文档淹没。解法是从简开始，只有在工具复杂度确实能带来帮助时再增加——避免"教条式"地照抄那些繁复的工作流，这种流程只增加过程本身、带不来价值。
False confidence is perhaps the subtlest pitfall. A passing spec test doesn't guarantee correct software—it only guarantees that the software matches the spec. If the spec is wrong, the code will faithfully implement the wrong thing. Specifications require the same careful review as code; they are not a silver bullet that eliminates the need for human judgment about requirements.
虚假信心或许是最隐蔽的陷阱。规约测试通过并不保证软件正确——它只保证软件与规约一致。如果规约是错的，代码就会忠实地实现那个错误。规约本身和代码一样需要被仔细评审；它不是那种"一劳永逸、让人再也不用对需求动脑子"的银弹。
X. SDD vs Traditional Design Documents
十、SDD 与传统设计文档
A natural question arises: how is SDD different from traditional High-Level Design (HLD) and Low-Level Design (LLD) documents that software engineering has always used? After all, HLD describes architecture, LLD details implementation, and requirements documents specify functionality. Aren't these already specifications?
一个自然的问题是：SDD 和软件工程一直在用的传统高层设计（HLD）、低层设计（LLD）文档有什么不同？毕竟，HLD 描述架构，LLD 详述实现，需求文档规定功能——它们不已经是"规约"了吗？
The answer is nuanced. Traditional design documents are specifications—the difference lies not in what is written, but in how it is used and whether it stays aligned with code. Traditional software engineering produces many specification-like artifacts: Software Requirements Specifications (SRS) for functional and non-functional requirements, High-Level Design documents for architecture and components, Low-Level Design documents for class diagrams and algorithms, and interface specifications for API contracts and IDL.
答案需要一点细分。传统的设计文档确实是规约——差别不在于写了什么，而在于它怎么被使用、以及它有没有和代码保持对齐。传统软件工程会产出许多"类规约"的产物：描述功能性和非功能性需求的软件需求规格说明（SRS），描述架构与组件的高层设计文档，描述类图与算法的低层设计文档，以及关于 API 契约与 IDL 的接口规约。
The problem is not the absence of specs—it's that they drift. By Sprint 3, the HLD is outdated. By release 2, the SRS no longer matches the product. The code becomes the de facto truth, and the documents become historical artifacts that nobody trusts or updates.
问题不在于"没有规约"——而在于它们会漂移。到第 3 个 Sprint 时，HLD 已经过时；到 2.0 版本时，SRS 已经与产品对不上。代码事实上成了真相，而这些文档则沦为没人相信、没人更新的历史档案。
Key Insight — The Core Difference: Traditional design documents are advisory—developers read them, then write code that hopefully matches. SDD specs are enforced—tests fail if code diverges, and in spec-as-source approaches, code is regenerated rather than manually edited.
核心洞见 —— 核心区别： 传统设计文档是"建议性"的——开发者读完它们，然后写出"但愿能匹配"的代码。SDD 的规约是"强制执行"的——一旦代码偏离规约，测试就会失败；而在规约即源的做法里，代码是重新生成出来的，而不是手工改的。
What SDD actually adds is threefold. First, executable specifications: traditional specs are read by humans, while SDD specs are executed as BDD scenarios, API contract tests, or model simulations—if code doesn't match, the build fails. Second, CI/CD integration: modern SDD embeds spec validation into continuous integration, checking every commit against the spec so drift is caught immediately rather than during quarterly reviews. Third, AI consumption: traditional design docs were written for human readers, while SDD specs are structured so AI coding assistants can consume them, generating code and tests from specs rather than guessing from vague prompts.
SDD 真正新增的东西其实有三点。第一，可执行的规约：传统规约是让人读的，而 SDD 规约以 BDD 场景、API 契约测试或模型仿真的形式被执行——一旦代码不匹配，构建就失败。第二，与 CI/CD 集成：现代 SDD 把规约验证嵌入到持续集成里，每次提交都对照规约做检查，让漂移被立刻发现，而不是在"季度评审"时才露出来。第三，AI 消费者：传统设计文档是为人类读者写的，而 SDD 规约的结构让 AI 编程助手也能消费它们，从规约出发生成代码和测试，而不是从模糊提示中瞎猜。
SDD is not a revolution—it's an evolution. The core insight (write specs first, let code derive from them) has been agile wisdom for decades. What's new is better tooling that makes executable specs practical, CI/CD maturity that enables automated enforcement, and AI as consumer where spec quality directly determines output quality. As Bryan Finster observed [3]: "SDD is not a revolution... it's just BDD with branding." But the branding serves a purpose: it reminds practitioners that specs should be authoritative, not advisory, and that modern tooling can enforce what was previously left to human discipline.
SDD 不是一场革命——它是一场演化。它的核心洞见（先写规约，让代码由它派生）几十年来一直是敏捷智慧的一部分。真正新的东西是：更好的工具让"可执行规约"变得切实可行，CI/CD 的成熟让自动化强制执行成为可能，以及"AI 作为消费者"——在这个新角色里，规约质量直接决定输出质量。正如 Bryan Finster 所说 [3]："SDD 并不是什么革命……它就是贴上新牌子的 BDD。"但这个新牌子有它的用处：它提醒从业者，规约应该是权威性的、而不是建议性的；而且现代工具链已经能够强制执行那些以前只能靠人类纪律去守住的东西。
XI. Relationship to Existing Practices
十一、与现有实践的关系
SDD is not a replacement for existing development practices—it builds on and extends them in the context of AI-assisted development.
SDD 并不是要取代现有的开发实践——它是在 AI 辅助开发的语境下，对这些实践的延伸与扩展。
Test-Driven Development (TDD) [11] is SDD at the unit level. Writing a test first is writing a micro-specification that defines expected behavior before implementation. SDD extends this thinking to higher levels—features, systems, and architectures—applying the same "specify first" discipline at broader scope.
测试驱动开发（TDD）[11] 就是单元层级的 SDD。先写测试，就是在实现之前写一个定义预期行为的"微规约"。SDD 把这种思维推广到更高层级——功能、系统和架构——在更宽的范围内应用同一种"先规约"的纪律。
Behavior-Driven Development (BDD) [12] is the most direct ancestor of modern SDD. Gherkin scenarios are executable specifications that bridge business requirements and technical implementation. What AI-assisted SDD tools add is assistance in generating code from those specs, accelerating the path from scenario to working software.
行为驱动开发（BDD）[12] 是现代 SDD 最直接的祖先。Gherkin 场景就是一种可执行规约，它在业务需求和技术实现之间架起桥梁。AI 辅助的 SDD 工具在它之上增加的，是"从这些规约生成代码"的辅助能力，把"从场景到可运行软件"的路径加速了。
Domain-Driven Design (DDD) aligns well with SDD through its emphasis on ubiquitous language—specs written in domain terms that both developers and stakeholders understand. The shared vocabulary that DDD advocates becomes the foundation for specifications that are meaningful to all parties.
领域驱动设计（DDD）与 SDD 非常契合，因为它强调"通用语言（ubiquitous language）"——用领域术语写出、开发者和利益相关方都能理解的规约。DDD 所倡导的共享词汇，正好成为"对各方都有意义的规约"的基础。
Agile methodologies are compatible with SDD. User stories with acceptance criteria are specifications; the Definition of Done is a form of spec. The difference is emphasis: SDD treats these artifacts as authoritative rather than advisory, and enforces alignment through automation rather than relying on human discipline alone.
敏捷方法论与 SDD 也是兼容的。带验收标准的用户故事就是规约；"完成的定义（Definition of Done）"也是一种规约。差别在于侧重点：SDD 把这些产物视为权威性的，而不是建议性的；并通过自动化来强制对齐，而不是单靠人的纪律。
XII. Conclusion
十二、结论
Spec-driven development inverts the traditional relationship between specifications and code. Instead of code being the source of truth with documentation as an afterthought, SDD makes specifications authoritative and code derivative. This inversion becomes increasingly important as AI coding assistants become more capable—when an AI can generate code from specifications faster than humans can type, the bottleneck shifts to specification quality.
规约驱动开发颠倒了规约与代码之间的传统关系。代码不再是事实之源、文档沦为事后补丁，而是：SDD 让规约成为权威，代码作为派生物。随着 AI 编程助手越来越强大，这种颠倒变得越来越重要——当 AI 从规约生成代码的速度，比人类打字还快时，瓶颈就转移到了规约质量上。
Specifications remove ambiguity for both human developers and AI assistants, preventing the guesswork and misinterpretation that leads to costly rework. The three levels of rigor—spec-first, spec-anchored, and spec-as-source—provide options that match different project needs, from lightweight initial clarity to rigorous code generation. Mature tooling exists across the spectrum, from BDD frameworks and API specification tools to AI-assisted SDD toolkits, making spec-first workflows practical today. Teams should match rigor to need, using the minimum specification discipline that removes ambiguity for their context rather than over-engineering process.
规约对人类开发者和 AI 助手同时消除了歧义，避免那些会导致代价高昂返工的"猜测"和"误解"。三种严格度层级——规约先行、规约锚定、规约即源——给出了适配不同项目需要的选项，从"轻量的初始清晰度"一直覆盖到"严格的代码生成"。在整个光谱上都已有成熟工具——从 BDD 框架、API 规约工具，到 AI 辅助的 SDD 工具包——这让"规约先行"的工作流在今天就具备了可行性。团队应该让严格度匹配需求，用"恰好能消除歧义"的最低规约纪律来应对自身情境，而不是在流程上过度工程化。
SDD builds on decades of TDD and BDD wisdom while adapting these practices for the AI era. The ideas are not new; what's new is the tooling and AI capabilities that make specs more powerful than ever. Work is being redefined as developers shift from manual coding to orchestrating specifications, reviewing AI outputs, and focusing on high-level design.
SDD 建立在几十年 TDD 和 BDD 智慧的基础之上，同时为 AI 时代调整了这些做法。思想本身并不新；新的是那些让规约比以往任何时候都更强大的工具和 AI 能力。开发者的工作正在被重新定义：从手工写代码，转向编排规约、评审 AI 输出、聚焦高层设计。
As software systems grow more complex and AI becomes more capable, the question shifts from "what code should I write?" to "what specification should I provide?" Teams that master spec-driven development will get more value from AI tools while maintaining the clarity and traceability that complex systems require. SDD offers a framework for answering that question systematically—making specifications, not code, the primary artifact of software development.
随着软件系统越来越复杂、AI 越来越强大，问题正在从"我该写什么代码？"转变为"我该提供什么规约？"。那些掌握了规约驱动开发的团队，将能在从 AI 工具获得更多价值的同时，保留复杂系统所要求的清晰度与可追溯性。SDD 提供了一套系统性回答这个问题的框架——让规约而不是代码，成为软件开发的主要产物。
