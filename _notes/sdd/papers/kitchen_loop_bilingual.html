The Kitchen Loop: User-Spec-Driven Development for a Self-Evolving Codebase

厨房循环（Kitchen Loop）：以"用户—规约"驱动的、可自我演化的代码库开发方法

Yannick Roy — 0xAgentKitchen@gmail.com — March 2026

Yannick Roy — 0xAgentKitchen@gmail.com — 2026 年 3 月

Abstract

摘要

Code production is now a commodity; the bottleneck is knowing what to build and proving it works. We present the Kitchen Loop, a framework for autonomous, self-evolving software built on a unified trust model: (1) a specification surface enumerating what the product claims to support; (2) "As a User × 1000", where an LLM agent exercises that surface as a synthetic power user at ∼1,000× human cadence; (3) Unbeatable Tests, ground-truth verification the code author cannot fake; and (4) Drift Control, continuous quality measurement with automated pause gates. We validate across two production systems over 285+ iterations, producing 1,094+ merged pull requests with zero regressions detected by the regression oracle (methodology in Section 6.1). We observe emergent properties at scale: multi-iteration self-correction chains, autonomous infrastructure healing, and monotonically improving quality gates. The primitives are not new; our contribution is their composition into a production-tested system with the operational discipline that makes long-running autonomous evolution safe.

写代码（code production）如今已是商品化的能力；真正的瓶颈在于知道该构建什么、并能证明它真的有效。本文提出厨房循环（Kitchen Loop）：一个面向自治式、可自我演化的软件构建框架，建立在统一的信任模型之上：(1) 一个规约面（specification surface），枚举出产品声称支持的所有能力；(2) "As a User × 1000"——由 LLM 智能体作为合成超级用户，以约 1,000× 人类节奏去使用这个规约面；(3) 不可被作弊的测试（Unbeatable Tests）——以代码作者无法伪造的事实之源（ground truth）进行验证；(4) 漂移控制（Drift Control）——对质量的连续度量加上自动暂停门。我们在两套生产系统、285+ 次迭代上完成了验证，共合并了 1,094+ 个 PR，回归预言机（regression oracle）未检出任何回归（方法论见 6.1 节）。我们观察到一些规模涌现的特性：跨多次迭代的自纠错链路、自治式的基础设施自愈，以及单调改进的质量门。这些原语并不新鲜；我们的贡献在于把它们组合成一套经过生产验证的系统，并具备让长时运行的自治演化保持安全所需的运维纪律。

1 Core Contributions

1 核心贡献

This paper makes four claims:

本文提出四点主张：

1. "As a User" vs. task-completion. Agentic systems should not just close tickets — they should systematically exercise a product's specification surface the way a user would. The Kitchen Loop is a production-tested framework for this, grounded in synthetic user journeys rather than isolated issues.

1. "作为用户" 而不是"完成任务"。 智能体系统不应仅止于关闭工单——它们应当像用户一样，系统性地使用产品的规约面。厨房循环就是为此而生、并经过生产验证的框架，扎根于合成的用户旅程，而不是孤立的 issue。

2. The unified trust model. Autonomous evolution requires a unified trust model. Three components interlock to make it safe: a specification surface (what the product claims to do), unbeatable tests (ground-truth verification the author can't fake), and a regression oracle with drift control and automatic pause gates.

2. 统一信任模型（unified trust model）。 自治式演化必须建立在统一的信任模型之上。三个组件咬合在一起才能保证安全：一个规约面（产品声称能做什么）、不可被作弊的测试（作者无法伪造的事实之源验证）、以及一个带漂移控制和自动暂停门的回归预言机。

3. Unbeatable tests. Correctness requires adversarial, multi-model review. Implementer-written tests are necessary but insufficient — in our deployments, 38 passing unit tests coexisted with complete feature failure. The Kitchen Loop enforces correctness through adversarial UAT gates (sealed test cards, fresh evaluator, zero context) and mandatory cross-model review: every PR is challenged by independent agents (Codex, Gemini, CodeRabbit) before merge. No output is accepted as-is from the model that wrote it.

3. 不可被作弊的测试（Unbeatable Tests）。 正确性需要对抗性的、多模型的复审。实施者自己写的测试是必要的、但不充分——在我们的部署里，曾出现过 38 个单元测试全部通过、但功能完全失效的情况。厨房循环通过对抗性的 UAT 门（封装好的测试卡、新启的评估者、零上下文）以及强制的跨模型复审来保证正确性：每个 PR 在合并前都要被独立的智能体（Codex、Gemini、CodeRabbit）挑战。任何输出都不会被它的作者模型直接接受。

4. Bounded production evidence. Across two production systems and 285+ iterations, the Kitchen Loop produced 1,094+ merged pull requests with zero regressions detected by the regression oracle (methodology in Section 4.1), monotonically improving quality gates (76–91% → 100%), and a cost of ∼$0.38 per merged PR.

4. 有界的生产证据。 在两套生产系统、285+ 次迭代中，厨房循环合并了 1,094+ 个 PR，回归预言机未检出任何回归（方法论见 4.1 节），质量门单调改进（76–91% → 100%），每个被合并的 PR 成本约 $0.38。

3 The Post-Commodity Code Thesis

3 后商品化代码论 (The Post-Commodity Code Thesis)

3.1 Code Is No Longer the Hard Part

3.1 代码不再是难点

LLM-based coding agents can produce functional code at a rate that renders the writing step non-limiting. Robbes et al. (2026) find that 15–23% of mature open-source projects adopted coding agents within just nine months of tool availability [4]. A senior engineer's value is shifting from writing code to knowing which code to write, why, and how to prove it's correct.

基于 LLM 的编码智能体如今的产出速度，已让"写代码"这一步不再是瓶颈。Robbes 等人（2026）发现，仅在工具问世九个月内，就有 15–23% 的成熟开源项目采用了编码智能体 [4]。资深工程师的价值正在转移：从"写代码"转向"知道该写哪段代码、为什么写、以及如何证明它正确"。

Yet the productivity story is more nuanced than adoption rates suggest. Becker et al. (2025) conducted the most rigorous randomized controlled trial to date: experienced open-source developers using AI copilots were 19% slower than those working without AI, contradicting both developer self-estimates (+20% speedup) and expert forecasts (+38% speedup) [12]. He et al. (2025) provide causal evidence that Cursor adoption produces a large but transient velocity boost (+281% lines added in month one, dissipating by month three) accompanied by persistent quality degradation: +30% static analysis warnings, +42% cognitive complexity, and +7% code duplication [11]. The accumulated complexity feeds back into velocity — every doubling of complexity reduces future velocity by ∼64.5% [11]. The Kitchen Loop can be read as a response to both findings: by shifting emphasis from code generation to specification and unbeatable verification, it addresses the quality degradation that erodes velocity gains and the lack of structural verification that leaves AI-assisted development slower than expected.

但生产力的故事比"采纳率"暗示的要复杂得多。Becker 等人（2025）做了至今最严谨的随机对照试验：使用 AI copilot 的资深开源开发者，比不使用 AI 的同伴慢了 19%——这与开发者自己的预估（+20% 提速）和专家的预测（+38% 提速）都背道而驰 [12]。He 等人（2025）从另一角度提供了因果证据：采用 Cursor 会带来一波很大但短暂的速度提升（首月新增代码行 +281%，到第三个月就消散了），同时伴随持续性的质量退化：静态分析告警 +30%、认知复杂度 +42%、代码重复 +7% [11]。累积的复杂度又会反噬速度——复杂度每翻倍，未来速度下降约 64.5% [11]。厨房循环可以视为对这两类发现的回应：通过把重心从"代码生成"转向"规约 + 不可被作弊的验证"，它直接处理那种侵蚀速度收益的质量退化，也补上了"AI 辅助开发反而更慢"背后所缺的结构化验证。

The same shift is happening to code review. Automated review tools can identify bugs, style violations, and security issues at a pace no human reviewer can match. The human reviewer's role shifts from "find the bug" to "judge whether this code serves the product's intent."

同样的转变也发生在 code review。自动化 review 工具识别 bug、风格违规和安全问题的速度，是任何人类 reviewer 都跟不上的。人类 reviewer 的角色从"找 bug"转向"判断这段代码是否服务于产品意图"。

This creates a new bottleneck: specification and verification. The hard problems are now: What should the product do? (Specification); Does the product actually do it? (Verification); Is the product getting better or worse over time? (Drift control).

这造就了新的瓶颈：规约（specification）与验证（verification）。如今的难题是：产品应该做什么？（规约）；产品有没有真的做到？（验证）；产品随着时间推移，是变好还是变差？（漂移控制）。

3.2 The Turing Test for Code

3.2 代码的图灵测试

Alan Turing's insight was that you don't need to understand how a machine thinks — you need a test rigorous enough that passing it constitutes sufficient evidence of capability. The machine can remain a black box. The test is the arbiter.

阿兰·图灵的洞见是：你不必理解机器是怎么思考的——你只需要一个足够严格的测试，一旦通过，就构成具备某种能力的充分证据。机器可以保持是黑盒，测试才是裁判。

We apply the same principle to AI-generated code. When an LLM agent writes a function, a connector, or a full feature, you face a choice: Option A — read every line, understand every branch, verify the logic manually (this scales poorly when the agent produces thousands of lines per day); Option B — build tests so rigorous that if the code passes them, you can trust it works without needing to audit every line. The code becomes a black box. The tests are the Turing Test.

我们把同样的原则应用到 AI 生成的代码上。当一个 LLM 智能体写下一个函数、一个 connector，乃至一整套功能时，你面前有两个选择：方案 A——读每一行、理解每个分支、手动核对逻辑（在智能体每天产出数千行代码时，这条路很难扩展）；方案 B——构造足够严格的测试，使得只要代码通过了它们，你就可以信任它能工作，而无需逐行审计。代码本身成了黑盒，测试就是它的图灵测试。

Empirical evidence supports both options coexisting in practice. Huang et al. (2025) find that 69% of professional developers carefully review every agentic change and 75% read every line of AI-generated code — but crucially, developers working on unfamiliar tasks let agents drive implementation while monitoring program outputs rather than reviewing code line-by-line [2]. This validates Option B for the regime where the Kitchen Loop operates: autonomous agents on structured tasks with rigorous output verification.

经验证据支持两种方案在实践中并存。Huang 等人（2025）发现，69% 的专业开发者会仔细 review 每一次智能体改动，75% 会逐行阅读 AI 生成的代码——但关键的是，在面对自己不熟悉的任务时，开发者会让智能体主导实现，转而通过监控程序输出来把关，而不再逐行审代码 [2]。这恰好印证了厨房循环所处的运行模式下方案 B 的合理性：在结构化任务上让自治智能体跑，并以严格的输出验证作为信任锚点。

The urgency for Option B is underscored by a growing QA crisis in AI-assisted development. Fawzy et al. (2025) find that 36% of practitioners using AI code generation skip quality assurance entirely, 18% place uncritical trust in AI output, and 10% delegate QA back to the same AI that wrote the code [1]. The result is predictable: 68% of practitioners characterize the output as "fast but flawed" [1]. Voluntary QA discipline is demonstrably insufficient; structural enforcement — tests that cannot be skipped — is the only reliable solution.

方案 B 的紧迫性，被 AI 辅助开发中日渐严重的 QA 危机进一步放大。Fawzy 等人（2025）发现，使用 AI 代码生成的实践者中，36% 完全跳过质量保证，18% 对 AI 输出无条件信任，10% 把 QA 又交给写代码的同一个 AI [1]。结果不出所料：68% 的实践者把这种产出形容为"快但有缺陷（fast but flawed）" [1]。仅凭自觉去做 QA 显然是不够的；只有结构化地强制——那种无法被跳过的测试——才是可靠的解。

Option B requires a specific kind of test: not unit tests written by the same agent that wrote the code (the agent could pass its own tests trivially), but end-to-end verification against real-world state. For a DeFi SDK, this means executing transactions on chain forks and verifying balance changes. For a signal platform, this means cross-referencing claims against live data APIs. For a web application, this means browser automation against a real rendering engine. For any product, it means testing at the boundary where the software meets reality.

方案 B 需要一种特定的测试：不是由写代码的同一个智能体写的单元测试（智能体可以轻易让自己的测试通过），而是端到端、对真实世界状态的验证。对一个 DeFi SDK 来说，意味着在链 fork 上执行交易并校验余额变化；对一个 signal 平台来说，意味着把信号声明拿去和实时数据 API 交叉核对；对一个 web 应用来说，意味着用浏览器自动化在真实渲染引擎上跑；对任何产品来说，都意味着在软件与现实接壤的边界上做测试。

We call these unbeatable tests — tests that verify outcomes against ground truth that the code author cannot fake. The test doesn't care how the code achieves the result — only that it does.

我们把这一类测试称作不可被作弊的测试（unbeatable tests）——依据代码作者无法伪造的事实之源去验证结果。测试不关心代码用什么手段达成结果——只关心结果是否达成。

It is of utmost importance to be aware of the weaknesses of LLMs when it comes to tests. There is a fallacy to believe that LLMs are so good at writing code, that they should be equally good at writing tests. They aren't. They often obsess over the test itself forgetting that passing the test is not the goal in itself. That obsession to pass the test often leads to cheating, modifying the test or even writing side scripts nailing the test, leaving the actual codebase untested.

特别要警惕 LLM 在测试方面的弱点。有一种谬误是：既然 LLM 这么会写代码，它写测试也应当一样厉害。事实并非如此。它们经常对"测试本身"过度执念，忘了"通过测试"并非目标本身。这种执念常常导向作弊：改测试、甚至写一个旁路脚本去单独击中那个断言，留下真正的代码库其实根本没被测过。

4 Related Work

4 相关工作 (Related Work)

The Kitchen Loop exists within a rapidly evolving landscape of agentic software engineering. We position our contributions against four categories of related work.

厨房循环身处一片快速演进的智能体软件工程版图之中。我们从四个相关方向定位本文的贡献。

4.1 Autonomous Coding Agents

4.1 自治式编码智能体 (Autonomous Coding Agents)

SWE-agent (Yang et al., 2024) and AutoCodeRover (Zhang et al., 2024) demonstrated that LLM agents can resolve GitHub issues autonomously by navigating codebases, editing files, and running tests. OpenHands (Wang et al., 2024) and Devin (Cognition, 2024) extended this to multi-step workflows including environment setup, debugging, and deployment. RepairAgent (Bouzenia et al., 2024) focused specifically on automated program repair with LLM-driven fault localization.

SWE-agent（Yang 等，2024）和 AutoCodeRover（Zhang 等，2024）证明了 LLM 智能体可以通过浏览代码库、编辑文件、运行测试，自治地解决 GitHub issue。OpenHands（Wang 等，2024）和 Devin（Cognition，2024）进一步把它扩展到包含环境搭建、调试、部署在内的多步工作流。RepairAgent（Bouzenia 等，2024）则专注于以 LLM 驱动故障定位的自动化程序修复。

These systems operate in task-completion mode: given an issue, produce a patch. The Kitchen Loop operates in coverage-exhaustion mode: given a specification surface, systematically exercise every combination until the gap between spec and reality approaches zero. The unit of work is not "resolve this issue" but "attempt this user scenario end-to-end and document everything that breaks." Task completion is a component of the loop (the Execute phase), not the loop itself.

这些系统都运行在任务完成（task-completion）模式：给一个 issue，产出一个 patch。厨房循环运行在覆盖耗尽（coverage-exhaustion）模式：给一个规约面，系统性穷举每一种组合，直到规约与现实之间的缺口趋近于零。工作单元不是"解决这个 issue"，而是"端到端尝试这个用户场景，并把所有出问题的地方记录下来"。任务完成只是循环里的一个组件（即 Execute 阶段），不是循环本身。

4.2 Self-Improving and Self-Evolving Agents

4.2 自我改进与自我演化的智能体 (Self-Improving and Self-Evolving Agents)

SICA ("A Self-Improving Coding Agent", arXiv 2504.15228, 2025) introduced agents that edit their own prompts and tools based on execution feedback, achieving self-improvement without human intervention. AlphaEvolve (DeepMind, 2025) demonstrated repository-scale code evolution using LLMs guided by automated evaluators. Confucius SDK proposed a meta-agent build-test-improve loop for SDK development.

SICA（《A Self-Improving Coding Agent》，arXiv 2504.15228，2025）提出了能基于执行反馈编辑自己 prompt 与工具的智能体，无需人工介入即可自我改进。AlphaEvolve（DeepMind, 2025）展示了由 LLM 在自动化评估器引导下进行仓库级代码演化。Confucius SDK 提出了用于 SDK 开发的元智能体 build-test-improve 循环。

The Kitchen Loop shares the self-improvement property (Section 13) — it discovers its own infrastructure bugs and updates its own skill files. The key difference is that self-improvement in the Kitchen Loop is anchored to a specification surface and regression oracle, not to an optimization objective. The loop doesn't optimize for a metric; it converges toward a specification. This prevents the "goodharting" failure mode where an agent optimizes a proxy metric while the product degrades in unmeasured dimensions.

厨房循环也具有自我改进的属性（见第 13 节）——它会发现自己基础设施里的 bug，并更新自己的 skill 文件。关键差异在于：厨房循环的自我改进锚定在规约面 + 回归预言机，而不是某个优化目标。循环不优化某个度量；它向规约收敛。这避免了"goodharting"的失效模式——智能体在某个代理指标上越做越漂亮，而产品在没有被度量的维度上其实在退化。

4.3 Long-Running and Looping Patterns

4.3 长跑型与循环型范式 (Long-Running and Looping Patterns)

Ralph Wiggum loops (Huntley, 2025; snarktank/ralph) demonstrated that autonomous agents can produce 1,000+ commits overnight in a continuous loop. Multi-agent SE frameworks like MetaGPT (Hong et al., 2024), ChatDev (Qian et al., 2024), and ALMAS assigned different agent roles (PM, architect, coder, tester) to simulate a software team.

Ralph Wiggum 循环（Huntley, 2025；snarktank/ralph）展示了自治智能体在一夜之间可以产生 1,000+ 个 commit 的连续循环。MetaGPT（Hong 等，2024）、ChatDev（Qian 等，2024）和 ALMAS 等多智能体软件工程框架，则给不同智能体分配 PM、架构师、coder、tester 等角色，模拟一个软件团队。

The Kitchen Loop draws from this lineage but adds three structural elements that Ralph-style loops lack: (1) a production-parity ticket management layer (Linear) where human and AI tickets are treated identically, preventing the loop from diverging from human priorities; (2) automated drift control with pause gates that halt the loop when quality degrades; and (3) the three-tier strategy model (Foundation/Composition/Frontier) that ensures balanced coverage rather than random or greedy scenario selection.

厨房循环承袭这一传统，但加入了 Ralph 风格的循环所缺的三个结构性元素：(1) 一个生产对等的工单管理层（Linear），其中人类工单和 AI 工单被同等对待，避免循环偏离人类的优先级；(2) 自动化漂移控制 + 暂停门，在质量退化时停下循环；(3) 三层策略模型（Foundation/Composition/Frontier），确保覆盖均衡，而不是随机或贪心地选场景。

4.4 Spec-Driven and Verification-Focused Approaches

4.4 规约驱动与重视验证的方法 (Spec-Driven and Verification-Focused Approaches)

SpecRover (Ruan et al., 2024) used natural-language specifications to guide automated debugging. AgileGen and Augmented Agile explored LLM-assisted requirements engineering and test generation from user stories. USEagent proposed user-story-driven end-to-end testing.

SpecRover（Ruan 等，2024）使用自然语言规约引导自动化调试。AgileGen 与 Augmented Agile 探索了 LLM 辅助的需求工程和从 user story 生成测试。USEagent 提出了由 user story 驱动的端到端测试。

The Kitchen Loop's AaU1000 method is closest to USEagent's vision but differs in scope and trust model. USEagent generates tests from user stories; AaU1000 generates usage scenarios from specification surfaces and validates them through unbeatable tests (4-layer verification against ground truth). The distinction matters: a test generated from a user story proves the story is implemented; a usage scenario validated through state-delta verification proves the product actually works in the way a real user would experience it.

厨房循环里的 AaU1000 方法在愿景上最接近 USEagent，但作用范围和信任模型不同。USEagent 从 user story 生成测试；AaU1000 从规约面生成使用场景，并用不可被作弊的测试（对事实之源做四层验证）去验证它们。这一区别是关键：从 user story 生成的测试只能证明"这个 story 被实现了"；通过状态差（state delta）验证过的使用场景，证明的是"产品确实按真实用户体验到的方式在工作"。

4.5 Vibe Coding and the QA Crisis

4.5 Vibe Coding 与 QA 危机

Fawzy et al. (2025) find a "speed-quality paradox" across 101 practitioner sources: 62% of vibe coders cite speed as their primary motivation, yet 68% characterize output as "fast but flawed" [1]. Huang et al. (2025) confirm that professional developers reject the vibe coding paradigm, carefully controlling agents through strategic plans — averaging only 2.1 execution steps per prompt despite plans spanning 70+ steps [2]. Their task suitability taxonomy identifies agents as unsuitable for business logic, domain knowledge, complex reasoning, and security-critical code — precisely the categories where the Kitchen Loop's spec surface and UAT gate provide structural guardrails.

Fawzy 等人（2025）在 101 份实践者资料中发现一个"速度—质量悖论"：62% 的 vibe coder 把速度列为首要动机，但 68% 又把产出形容为"快但有缺陷" [1]。Huang 等人（2025）证实，专业开发者拒绝 vibe coding 范式：他们用战略计划严格控制智能体——尽管计划往往跨越 70+ 步，每个 prompt 平均只让智能体执行 2.1 步 [2]。他们的任务适配性分类法指出，智能体不适合处理业务逻辑、领域知识、复杂推理和涉及安全的代码——而恰恰这些类别，正是厨房循环的规约面与 UAT 门提供结构性护栏的地方。

4.6 Benchmark Contamination, Adoption, and Quality Evidence

4.6 基准污染、采纳度与质量证据 (Benchmark Contamination, Adoption, and Quality Evidence)

The dominant evaluation benchmark, SWE-Bench, suffers from data contamination: Liang et al. (2025) show a 23-percentage-point accuracy gap between memorized and novel repositories [6], while Thai et al. (2025) find GPT-5 scores 65% on single-issue tasks but only 21% on multi-file evolution tasks [7]. Meanwhile, adoption is accelerating — Robbes et al. (2026) find 15–23% of mature open-source projects adopted coding agents within nine months [4] — but quality lags: 90.6% of agent-authored PRs receive zero human review [3], and ∼40% of Copilot-generated code contains security vulnerabilities [10]. The Kitchen Loop addresses both problems: it rejects benchmark-style evaluation in favor of verification against live product state, and interposes adversarial verification between generation and merge.

主流评测基准 SWE-Bench 存在数据污染问题：Liang 等人（2025）展示了"被记住的仓库"与"全新仓库"之间存在 23 个百分点的准确率差距 [6]；而 Thai 等人（2025）发现，GPT-5 在单 issue 任务上得分 65%，但在多文件演化任务上只有 21% [7]。与此同时，采纳度在加速——Robbes 等人（2026）发现，仅九个月内，就有 15–23% 的成熟开源项目采用了编码智能体 [4]——但质量却跟不上：90.6% 的智能体撰写的 PR 没有得到任何人类 review [3]，约 40% 的 Copilot 生成代码包含安全漏洞 [10]。厨房循环正面回应了这两个问题：它拒绝基准式评测，转而对实时产品状态做验证；并在生成与合并之间插入对抗性的验证。

4.7 Multi-Agent Design Patterns

4.7 多智能体设计模式 (Multi-Agent Design Patterns)

Cai et al. (2025) systematically review 94 LLM-based multi-agent systems for software engineering, identifying 16 design patterns ranked by adoption frequency [8]. The Kitchen Loop composes seven of these patterns into a single integrated loop: Role-Based Cooperation (six-phase loop with distinct roles per phase), Self-Reflection (Polish and Regress phases), Cross-Reflection (UAT Gate), Debate-Based Cooperation (Discussion Manager), Voting-Based Cooperation (multi-model tribunal), Tool-Agent Registry (skills directory), and Agent Evaluator (regression oracle). Notably, three of these — Debate-Based Cooperation (4.3% adoption), Voting-Based Cooperation (3.2%), and Agent Evaluator (3.2%) — are among the least adopted patterns in the literature [8], suggesting the Kitchen Loop operationalizes architectural ideas the field has not yet widely implemented.

Cai 等人（2025）系统综述了 94 个面向软件工程的 LLM 多智能体系统，按采纳频率列出 16 种设计模式 [8]。厨房循环把其中 7 种模式组合到同一个集成循环里：基于角色的协作（六阶段循环、每个阶段角色分明）、自反思（Polish 与 Regress 阶段）、交叉反思（UAT 门）、基于辩论的协作（Discussion Manager）、基于投票的协作（多模型陪审团）、Tool-Agent 注册表（skills 目录）、智能体评估者（回归预言机）。值得注意的是，其中三种——基于辩论的协作（4.3% 采纳率）、基于投票的协作（3.2%）、智能体评估者（3.2%）——都是文献里采纳率最低的模式 [8]，这暗示厨房循环把领域里尚未被广泛实现的架构想法，落到了可运行的层面。

5 The "As a User x 1000" Method (AaU1000)

5 "作为用户 × 1000"方法 (AaU1000)

5.1 The Core Insight

5.1 核心洞察

The most reliable signal about whether a feature works is a real usage attempt — not a unit test, not a code review, but someone exercising the product end-to-end. The problem: human usage attempts are slow and expensive. The AaU1000 method replaces the "someone" with an LLM agent that:

判断一个功能是否真的工作，最可靠的信号是一次真实的使用尝试——不是单元测试，不是 code review，而是有人端到端地把产品用一遍。问题在于：人类的使用尝试又慢又贵。AaU1000 方法把这里的"有人"替换成一个 LLM 智能体，由它来：

1. Selects a realistic usage scenario derived from the product's specification surface 2. Attempts it as a real user would — writing code, running it, observing what breaks 3. Documents failures as actionable tickets — not vague observations but precise bug reports with reproduction steps, root cause hypotheses, and file-level pointers 4. Fixes those failures immediately — implementing the fix, writing tests, and shipping a PR 5. Watches for regression and drift — verifying nothing else broke and the codebase is not getting worse

1. 从产品的规约面里挑选一个真实的使用场景； 2. 像真实用户那样尝试它——写代码、跑代码、观察哪里崩溃； 3. 把失败记录成可执行的工单——不是模糊的吐槽，而是带复现步骤、根因假设、文件级指针的精确 bug 报告； 4. 立即修复这些失败——实现修复、写测试、提 PR； 5. 关注回归与漂移——确认没有别处出问题、且代码库没有变得更差。

And then does it again. At whatever cadence the infrastructure allows.

然后，再来一次。以基础设施允许的最快节奏。

5.2 Quantifying the 1000x

5.2 把 1000× 量化

The "1000x" is an order-of-magnitude claim, not a precise multiplier. We ground it empirically:

"1000×"是一个数量级量级（order-of-magnitude）的说法，而不是精确倍数。我们用经验数据给它一个落脚点：

Single-thread velocity: A senior engineer realistically ships ∼15-25 merged PRs per month (bug fixes, features, tests). Each Kitchen Loop instance runs single-threaded per product. The strategy framework produced 728+ merged PRs in ∼5 weeks (∼145/week); the signal platform produced 366 in 17 days (∼150/week). Per-system, this is a ∼24-48x single-thread throughput increase depending on domain iteration speed (the signal platform iterates ∼25x faster than the SDK). Combined across both systems running concurrently, total output was 1,094+ merged PRs.

单线程速度。 一位资深工程师每月现实可合并的 PR 大约是 15–25 个（含 bug 修复、新功能、测试）。每个厨房循环实例对一个产品来说是单线程跑的。Strategy 框架在大约 5 周里产出 728+ 个合并的 PR（约 145/周）；signal 平台在 17 天里产出 366 个（约 150/周）。按系统计算，这是单线程吞吐量的约 24–48 倍提升，具体取决于领域的迭代速度（signal 平台比 SDK 快约 25 倍）。两套系统并发运行下，合计 1,094+ 个合并 PR。

Scenario throughput: The DeFi strategy framework's loop completes one full usage scenario (ideate → implement → test → fix → regress) in 80-230 minutes. A human attempting the same scenario takes 1-3 days. This is a 7-25x per-scenario speedup.

场景吞吐。 DeFi strategy 框架的循环完成一次完整使用场景（ideate → 实现 → 测试 → 修复 → 回归）需要 80–230 分钟。一个人类做同样的事要 1–3 天。这是每场景 7–25 倍的加速。

Parallelization potential: The loop currently runs single-threaded per product. Running N loops on N products (as demonstrated with SDK + Edge concurrently) scales linearly. With parallel execution infrastructure (dynamic port allocation, containerized test environments), a single product could run N loops exploring different specification regions simultaneously. The 1000x represents the achievable ceiling with moderate parallelization — not the current single-thread reality.

并行化潜力。 当前循环对每个产品是单线程跑的。在 N 个产品上跑 N 个循环（如 SDK + Edge 并发那样）是线性扩展。如果再加上并行执行基础设施（动态端口分配、容器化测试环境），同一个产品也可以跑 N 个循环并发探索规约面的不同区域。"1000×"代表的是适度并行下可达的天花板——不是当前单线程下的现实。

5.3 Spec-Driven, Not Spec-Speculative

5.3 规约驱动，而不是规约臆测 (Spec-Driven, Not Spec-Speculative)

The method is scoped to what exists and is expected to work today — not speculative feature engineering, benchmarking, or user research.

本方法的范围限定在"今天已存在、且预期能正常工作"的能力——不做投机性的功能设想、不刷基准、不做用户研究。

5.4 The "Obviously Missing" Signal

5.4 "明显缺失"信号

The most important class of findings is the "obviously missing" signal: features any competent user would expect, but that don't work. These are objectively verifiable failures — a function returning AttributeError, a missing configuration entry, a gas cap blocking all transactions on a chain. The loop discovers verifiable failures, not speculative issues.

这里最重要的一类发现是"明显缺失（obviously missing）"信号：任何称职用户都会期待存在、但实际上不工作的功能。这些是客观可验证的失败——一个返回 AttributeError 的函数、一个缺失的配置项、一个把整条链上所有交易都卡死的 gas 上限。循环发现的是可验证的失败，而不是猜测性的疑虑。

5.5 The Three-Tier Strategy Model

5.5 三层策略模型 (Three-Tier Strategy Model)

The Kitchen Loop formalizes a three-tier scenario generation model that ensures balanced coverage across the product's maturity spectrum:

厨房循环把场景生成形式化成一个三层模型，确保覆盖在产品成熟度光谱上保持均衡：

Tier 1 — Foundation (30%) exercises a single feature on a single platform or configuration. One integration, one action, happy path. These scenarios should be trivially achievable by a new user in a few minutes. If anything goes wrong, it is a critical regression. Foundation iterations maintain the baseline: the easy stuff must always be bulletproof.

Tier 1 — Foundation（30%） 在单一平台或单一配置上行使单一功能。一个集成、一个动作、一条 happy path。这些场景应该是新用户几分钟内就能轻松走通的。如果出错，那就是严重回归。Foundation 迭代守的是基线：简单的事必须始终铁桶不漏。

Tier 2 — Composition (50%) combines two or more features in creative ways. Multi-service flows, indicator-driven behavior, multi-step workflows, cross-platform configuration stress tests. The goal is to find bugs at the seams between components that pass individually but fail in combination. This is where the majority of novel bug discovery happens, because the combinatorial space of feature pairs and triples is far larger than the space of individual features.

Tier 2 — Composition（50%） 把两个或更多功能以创造性方式组合起来。多服务流、由指标驱动的行为、多步工作流、跨平台配置压力测试。目标是揪出那些组件单独都通过、但组合起来出错的"接缝处"bug。这是大部分新 bug 被发现的地方——因为功能两两、三三组合的空间，远比单功能空间大得多。

Tier 3 — Frontier (20%) deliberately reaches beyond the product's current capabilities. The deliverable shifts from "working scenario" to "gap analysis": what would you need to build this? What's missing? What would it unlock? The experience report emphasizes specific missing features, ordered by implementation effort and user value.

Tier 3 — Frontier（20%） 故意去触碰产品当前能力之外的边界。交付物从"可用场景"变成"缺口分析"：要做出这个，你还需要建什么？少了什么？做出来能解锁什么？体验报告的重点放在具体的缺失功能上，按实现成本和用户价值排序。

5.6 The Self-Expanding Property

5.6 自扩张属性 (Self-Expanding Property)

The three tiers form a self-reinforcing growth cycle. This is the "secret sauce" that transforms a random agent into a strategic engineer:

三个 tier 共同构成一个自我强化的成长循环。这是把"随机的智能体"变成"战略级工程师"的"秘方"：

Iteration N: Features = {A, B}
  T1 (Foundation): test A, test B (2 scenarios)
  T2 (Composition): test A+B, test B+A (2 scenarios)
  T3 (Frontier): "I need C to do A+C" → gap analysis (1 gap report)

— C gets built —

Iteration N+k: Features = {A, B, C}
  T1 (Foundation): test A, test B, test C (3 scenarios)
  T2 (Composition): test A+B, A+C, B+C, A+B+C (4+ scenarios)
  T3 (Frontier): "Now I want D to do A+C+D" (1 gap report)
迭代 N: 已有功能 = {A, B}
  T1 (Foundation): 测 A、测 B（2 个场景）
  T2 (Composition): 测 A+B、测 B+A（2 个场景）
  T3 (Frontier): "我需要 C 才能做 A+C" → 缺口分析（1 份缺口报告）

— C 被建出来 —

  T1 (Foundation): 测 A、测 B、测 C（3 个场景）
  T2 (Composition): 测 A+B、A+C、B+C、A+B+C（4+ 个场景）
  T3 (Frontier): "现在我想要 D 来做 A+C+D"（1 份缺口报告）

Foundation grows linearly with each new feature. Composition grows superlinearly — each new feature can be combined with every existing feature and pair. Frontier pushes the boundary outward. The loop evolves the product by systematically testing what exists and identifying the next most valuable capability.

每加入一个新功能，Foundation 线性增长。Composition 则超线性增长——每个新功能都能与所有已有功能、所有已有 pair 组合。Frontier 把边界向外推。循环就是这样系统性地测试已有能力、并识别下一个最有价值的能力来演化产品。

Design Principles

设计原则

P1. Ground-Truth Verification. Trust in AI-generated code should be proportional to how ground-truth-verifiable the test outcomes are, not to test coverage percentage.

P1. 事实之源验证 (Ground-Truth Verification)。 对 AI 生成代码的信任，应当与测试结果"可被事实之源验证（ground-truth-verifiable）"的程度成正比，而不是与测试覆盖率成正比。

P2. Weakest-Evaluator. A user test is only trustworthy if a minimally capable evaluator can execute it. (§4.8)

P2. 最弱评估者 (Weakest-Evaluator)。 一个用户测试的可信度，取决于一个能力最低的评估者是否能跑通它（参见 §4.8）。

P3. Spec-Anchored Improvement. Self-improving agents should optimize toward specification satisfaction, not proxy metrics. (§2.2, §3.5)

P3. 规约锚定的改进 (Spec-Anchored Improvement)。 自我改进的智能体应当向"满足规约"优化，而不是向某个代理指标优化（参见 §2.2、§3.5）。

P4. Drift-Before-Failure. Autonomous loops need continuous trend monitoring, not only binary quality gates. (§5.5)

P4. 在崩盘之前察觉漂移 (Drift-Before-Failure)。 自治循环需要的是连续的趋势监控，而不仅仅是二值的质量门（参见 §5.5）。

6 Unbeatable Tests: The Multi-Tier QA Framework

6 不可被作弊的测试：多层 QA 框架 (Unbeatable Tests: The Multi-Tier QA Framework)

6.1 Methodology: What "Zero Regressions" Means

6.1 方法论：什么叫"零回归"

Throughout this paper, we report "zero regressions introduced by loop-merged code." This claim requires precise definition to be falsifiable: Detection method — every iteration ends with an automated regression oracle run (Section 5.2). For the DeFi strategy framework, this means executing demo strategies on Anvil chain forks and verifying 4-layer state deltas. For the signal platform, this means running the full quality-gate pipeline (structural, factual, temporal, cognitive) plus anti-signal canaries. Measurement window — continuous, every iteration, not sampled. The oracle runs after every merge, not on a periodic schedule. Scope — the claim covers regressions detected by the oracle in code merged through the loop's own process. It does not claim the codebase is bug-free, nor that latent regressions undetectable by the oracle do not exist. The oracle's coverage is bounded by its test suite (10,913 unit tests, 62 demo strategies, 77 signal verifiers). Exclusions — environmental failures (API timeouts, chain fork instability) are distinguished from code regressions by correlation with recent merges. A failure that reproduces on the pre-merge commit is environmental; a failure introduced by a specific PR is a regression.

全文中我们都在说"循环合并的代码引入零回归"。要让这个说法可证伪，必须有精确定义。检测方法——每次迭代都以一次自动化回归预言机运行结尾（§5.2）。对于 DeFi strategy 框架，意味着在 Anvil 链 fork 上执行 demo 策略，并校验四层状态差。对于 signal 平台，意味着跑完整的质量门流水线（结构、事实、时序、认知）外加反信号哨兵（anti-signal canaries）。度量窗口——连续度量，每次迭代都做，不做采样。预言机在每次合并后都跑，而不是按周期跑。范围——这个声明仅覆盖"循环自身流程合并的代码"中、被预言机检出的回归。它并不声称代码库里没有 bug，也不声称不存在预言机检不出的潜在回归。预言机的覆盖能力以其测试套件为界（10,913 个单元测试、62 个 demo 策略、77 个 signal 验证器）。排除项——环境性失败（API 超时、链 fork 不稳）通过"是否与最近合并相关"来与代码回归区分开。一个失败若在合并前的 commit 上也能复现，那就是环境问题；只在某个 PR 之后才出现的失败，才是回归。

This definition makes the claim auditable: any third party with access to the oracle suite and git history can reproduce the measurement.

这种定义让该说法可被审计：任何第三方只要拿到预言机套件和 git 历史，都能复现这个度量。

6.2 Why Tests Are the New Competitive Advantage

6.2 为什么测试是新的竞争优势

Gao et al. (2025) find functional bugs in 78% of studies on AI-generated code, with ∼40% of Copilot code containing security vulnerabilities [10]. Dominant benchmarks (HumanEval, MBPP) rely on Pass@k metrics that overlook semantic correctness [10], and Liang et al. (2025) show SWE-Bench performance is inflated by data contamination (76% accuracy on memorized paths vs. 53% on novel repositories) [6]. Unbeatable tests sidestep both failure modes: they verify against live product state that changes with every deployment, making both contamination and pass-rate gaming irrelevant.

Gao 等人（2025）的综述发现，关于 AI 生成代码的研究中有 78% 报告了功能性 bug，约 40% 的 Copilot 代码包含安全漏洞 [10]。主流基准（HumanEval、MBPP）依赖 Pass@k 之类的指标，对语义正确性视而不见 [10]；Liang 等人（2025）则指出 SWE-Bench 的成绩被数据污染拉高（被记住的路径上 76%，全新仓库上只有 53%）[6]。不可被作弊的测试同时绕过了这两类失效：它依据每次部署都会变化的实时产品状态做验证，使数据污染和"刷通过率"都失效。

This requires tests at multiple tiers, each catching a different class of defect that lower tiers miss.

而要做到这一点，需要多层测试，每层都能拦截下层漏掉的一类缺陷。

The critical insight: L1 and L2 are necessary but not sufficient. 6.4 The 4-Layer Verification Pattern

6.4 4 层验证模式 (The 4-Layer Verification Pattern)

Every L3 integration test implements four verification layers, each catching a different class of defect. We illustrate with the DeFi strategy framework (Case Study A, Section 10); other domains substitute their own verification layers (e.g., browser automation for web apps, API contract testing for backends — see Section 16 for adaptation examples).

每个 L3 集成测试都实现四层验证，每层拦截不同类别的缺陷。下面以 DeFi strategy 框架（案例 A，§10）为例说明；其他领域按各自情况替换验证层（如 web 应用换成浏览器自动化、后端换成 API 契约测试——见 §16 的适配示例）。

A test that only compiles is incomplete. A test that compiles and executes but doesn't check state deltas is dangerously incomplete — it could silently succeed while doing the wrong thing. Layer 4 is what makes the test unbeatable: it verifies the outcome, not just the execution.

只编译就过的测试是不完整的。能编译能执行、但不检查状态差的测试是危险的不完整——它可能在静默地"做错事"的同时还显示成功。第 4 层才是让测试真正不可被作弊的关键：它验证的是结果，不只是执行。

For failure-mode tests: 3 layers are required (compilation, execution, state deltas), and state conservation MUST be asserted (state before and after remain unchanged). The system must not silently lose assets when it fails.

对失效模式（failure-mode）测试：必须有 3 层（编译、执行、状态差），并且必须断言状态守恒（前后状态保持不变）。系统失败时不得静默地丢失资产。

6.5 The Coverage Matrix: Exhaustive by Design

6.5 覆盖矩阵：穷举式设计 (The Coverage Matrix: Exhaustive by Design)

The specification defines a coverage matrix: every combination of feature, platform, and action type that the product claims to support. The test suite's goal is to fill every cell in this matrix.

规约定义了一个覆盖矩阵：产品声称支持的每一种"功能 × 平台 × 动作类型"组合。测试套件的目标就是把矩阵的每一格填满。

For a product with N features, M platforms, and K action types, the matrix has N x M x K cells. Each cell represents a claim: "Feature X works on Platform Y for Action Z." Each empty cell is an untested claim — a place where the product might silently fail.

对于 N 个功能、M 个平台、K 种动作类型的产品，矩阵有 N × M × K 格。每一格代表一个声明："功能 X 在平台 Y 上对动作 Z 有效"。每一个空格都是一个未被测试的声明——也就是产品可能悄悄出错的地方。

The Kitchen Loop fills these cells systematically, prioritizing by risk: P0 — core features on primary platforms (table stakes); P1 — core features on secondary platforms (breadth); P2 — advanced features on primary platforms (depth); P3 — edge cases and aggregators (completeness).

厨房循环按风险优先级，系统性地把这些格子填上：P0——核心功能在主平台上（基本盘）；P1——核心功能在次要平台上（广度）；P2——高级功能在主平台上（深度）；P3——边缘场景与聚合器（完整性）。

No manual QA process can fill a matrix with hundreds or thousands of cells on a weekly cadence. The Kitchen Loop can, because it generates and runs tests at AI speed.

任何手工 QA 流程都不可能以周为节奏填一个有数百乃至数千格的矩阵。厨房循环可以——因为它以 AI 速度生成并跑测试。

6.7 Multi-Model Review Tribunals

6.7 多模型复审陪审团 (Multi-Model Review Tribunals)

For decisions that require judgment (architectural choices, ambiguous test results, code quality assessment), the Kitchen Loop uses multi-model tribunals: three independent AI reviewers evaluate the same artifact in parallel. Findings are synthesized with consensus classification: Consensus (all three agree) treated as a confirmed finding; Majority (two agree) treated as a likely finding, prioritized for action; Solo (one reviewer) flagged for human judgment.

对那些需要判断的决策（架构选择、模糊的测试结果、代码质量评估），厨房循环采用多模型陪审团：三个独立 AI reviewer 并行评估同一份产物。结论按共识做分类：Consensus（三人都同意）视为已确认的发现；Majority（两人同意）视为可能成立的发现，优先处理；Solo（仅一人提出）标记为需要人类判断。

This reduces the false-positive rate of any single model and provides higher-confidence assessments for critical decisions. Section 7 extends this concept into a full structured deliberation system — the Discussion Manager — with multi-round debate, epistemic safeguards against sycophancy, and empirical validation across 23 production discussions.

这降低了任一模型的误报率，并为关键决策提供更高置信度的评估。第 7 节把这个概念扩展为完整的结构化审议系统——Discussion Manager——带多轮辩论、抗谄媚（anti-sycophancy）的认知防护，以及在 23 次生产讨论上的实证验证。

6.8 The Adversarial UAT Gate: "How Would a User Test This?"

6.8 对抗性 UAT 门："用户会怎么测它？" (The Adversarial UAT Gate)

Unit tests prove code works; they don't prove features work for users. When the same agent that implements a feature also tests it, three failure modes emerge: Happy-path blindness — the implementer only tests the case they built for; Context leakage — the agent "knows" the implementation and unconsciously compensates for gaps; Cheating — AI models optimize for green checks, not product truth. They write side scripts, mock data, and reinterpret assertions to force a pass.

单元测试证明的是"代码在跑"，并不证明"功能对用户有效"。当实现该功能的同一个智能体也来测它时，会出现三种失效：Happy-path 盲点——实现者只测了自己针对的那条路径；上下文泄漏——智能体"知道"实现细节，无意识地替缺口打了补丁；作弊——AI 模型优化的是"绿勾勾"，而不是产品真相。它们会写旁路脚本、mock 数据、重新解释断言，硬把测试搞绿。

The solution is not "ask the model to test honestly" but to design the loop so honesty is the easiest behavior and cheating is mechanically visible. The UAT Gate implements this through a three-step adversarial protocol:

解法不是"请模型诚实地测"，而是把循环设计成：诚实是最容易的行为，作弊机械上可见。UAT 门通过一个三步对抗协议来落地这个目标：

6.9 Step 1: "How would a user test this?"

6.9 第 1 步："用户会怎么测它？"

After implementing a ticket and creating a PR, the implementing agent must write a sealed test card — a step-by-step recipe that any user could follow to verify the feature works. The test card format is strict: every step has an exact command (no placeholders requiring judgment); every step has an exact expected exit code and exact output assertions (not "should work" — specific strings); at least one step verifies that bad input is rejected (not just happy path); no manual code edits — if testing requires configuration, the implementer ships a fixture in the PR; no implementation details — user-visible behavior only.

实现工单并提交 PR 之后，实施者智能体必须写一份封装测试卡（sealed test card）——一份任何用户都能照着走、用来验证功能是否有效的分步配方。测试卡格式严格：每一步都要有精确命令（不准用需要判断的占位符）；每一步都要有精确的预期退出码和精确的输出断言（不是"应该能行"，而是具体字符串）；至少有一步验证错误输入会被拒绝（不仅测 happy path）；不得手工改代码——如果测试需要配置，实施者要在 PR 里附上 fixture；不得包含实现细节——只描述用户可见行为。

This is the "as a user, how can I test this?" forcing function. If the implementer cannot write a card that demonstrates the feature working end-to-end, that's a signal the feature isn't done — even if all unit tests pass.

这是一个"作为用户，我怎么测"的强制函数。如果实施者根本写不出一张能端到端展示功能工作的测试卡，那就是这个功能其实没做完的信号——哪怕所有单元测试都绿。

6.10 Step 2: "Clear session — remove biases"

6.10 第 2 步："清空会话——抹掉偏见"

A fresh agent is spawned with zero implementation context to execute the test card: Information wall — the evaluator receives only the test card. No diff, no ticket, no code context, no conversation history from the implementing agent. Weakest model — the evaluator uses the weakest available model (e.g., Haiku) as a "dumb user" proxy. A strong model compensates for bad test cards by "figuring out" what the feature does. A weak model fails if the card is incomplete — which is exactly the signal you want. If the weakest model can follow the card and get expected results, a real user can too. Isolated worktree — the evaluator runs in a clean git worktree, a separate copy of the repository with pristine state. Read-only mandate — the evaluator is explicitly forbidden from editing any product file. Its prime directive is to disconfirm the feature, not confirm it.

启动一个新的智能体来执行测试卡，给它零实现上下文：信息隔墙——评估者只拿到测试卡，不看 diff、不看工单、不看代码上下文，也不看实施者智能体的对话历史。最弱模型——评估者用当前可用的最弱模型（如 Haiku）作为"笨用户"代理。一个强模型会替糟糕的测试卡"猜"出功能意图。一个弱模型则在测试卡不完整时直接失败——这正是你想要的信号。如果最弱的模型都能跟着测试卡走出预期结果，那真实用户也行。隔离 worktree——评估者跑在干净的 git worktree 上，是仓库的一个独立副本，状态原始。只读授权——明确禁止评估者编辑任何产品文件。它的首要任务是证伪这个功能，而不是证实它。

6.11 Step 3: "Do it — show me the results"

6.11 第 3 步："去做——把结果给我看"

The evaluator executes every step and produces structured evidence: raw command output, exact exit codes, actual-vs-expected comparisons. Then the implementing agent performs a mechanical integrity check:

评估者执行每一步，产生结构化证据：原始命令输出、精确退出码、实际值与预期值对比。然后实施者智能体执行机械式的完整性检查：

Three-Layer Anti-Cheating. (1) Information wall: evaluator gets ONLY the test card. No diff, no ticket, no implementation context. (2) Mechanical integrity check: after evaluation, git diff on the UAT worktree. Any product file modification = EVAL_CHEAT_FAIL. Any untracked files outside evidence dir = FAIL. (3) Evidence structure: every step requires raw command output + exit codes. Missing steps = failure. "It worked" without output = failure.

三层防作弊。 (1) 信息隔墙：评估者只拿到测试卡。无 diff、无工单、无任何实现上下文。(2) 机械完整性检查：评估之后，对 UAT worktree 跑 git diff。任何对产品文件的修改 = EVAL_CHEAT_FAIL。证据目录之外有任何未跟踪文件 = FAIL。(3) 证据结构：每一步都要有原始命令输出 + 退出码。缺步 = 失败。"It worked" 而无输出 = 失败。

6.12 Real-World Example: The Backtest Service Gap

6.12 真实案例：Backtest 服务的缺口

This example illustrates exactly why the UAT gate exists.

这个例子精确地说明了 UAT 门为什么必要。

An agent implemented a backtest HTTP service with 38 passing unit tests covering HTTP routing, job lifecycle state machines, model serialization, and capacity limits. All tests green. Lint passes. PR created. Ticket moved to "In Review."

一个智能体实现了一个 backtest 的 HTTP 服务，写了 38 个通过的单元测试，覆盖 HTTP 路由、作业生命周期状态机、模型序列化、容量上限。所有测试全绿，lint 也过了，PR 已建，工单挪到了 "In Review"。

But nobody ever actually started the service and sent a request to it.

但从来没有任何人真的启动过这个服务、向它发过一个请求。

The real smoke test — curl -X POST http://localhost:8000/api/v1/backtest followed by polling for completion — would have revealed that PnLBacktester() was instantiated with no constructor arguments. The data providers, fee models, and slippage models were never wired up. The service accepted jobs and immediately failed them.

真正的冒烟测试——curl -X POST http://localhost:8000/api/v1/backtest，再轮询作业完成状态——会立刻暴露：PnLBacktester() 在实例化时没传任何构造参数。数据源、手续费模型、滑点模型，根本没接上。服务收下作业后立刻把它们标成失败。

38 unit tests passed. The feature was completely broken.

38 个单元测试通过。功能完全坏了。

With the UAT gate, the implementer must write a test card whose steps include starting the service, submitting a job, polling for completion, and verifying results contain data. The Haiku evaluator runs this card blindly. Step 3 returns "status": "failed". Verdict: PRODUCT_FAIL. The ticket stays open until the agent actually wires up the backtest pipeline.

加上 UAT 门后，实施者必须写一张测试卡，步骤里要包含"启动服务、提交一个作业、轮询完成、并验证结果里有数据"。Haiku 评估者盲跑这张卡。第 3 步返回 "status": "failed"。判定：PRODUCT_FAIL。工单保持打开，直到智能体真的把 backtest 流水线接上。

The implementer can't dodge this: can't write "run the unit tests" as the card — validation rejects it (not user testing); can't test only HTTP routing — the card rules require testing the actual user journey; can't weaken assertions — "Expected output contains: completed" is binary; can't skip the gate — mandatory for any change to product code. In autonomous loops where 90.6% of agent-authored PRs receive zero human review [3], the UAT gate fills this gap mechanically.

实施者无法绕过：不能把"跑单元测试"写进卡里——校验会拒绝（这不是"用户测试"）；不能只测 HTTP 路由——卡的规则要求测真实的用户旅程；不能弱化断言——"预期输出包含: completed" 是二值的；不能跳过这道门——任何对产品代码的改动都强制要走。在那种 90.6% 的智能体撰写 PR 都没人 review 的自治循环里 [3]，UAT 门以机械方式补上了这块缺口。

7 Controlling Regression and Drift

7 控制回归与漂移

7.1 The Drift Problem

7.1 漂移问题

A self-evolving codebase faces a unique risk: quality drift. Each iteration produces code that passes its own tests — but does the accumulation of changes make the system better or worse? Without continuous measurement, the answer is unknowable until a user hits a regression in production.

自我演化的代码库面临一个独特的风险：质量漂移（quality drift）。每次迭代产出的代码都通过了它自己的测试——但所有变更累积起来，是让系统变好了还是变坏了？没有持续度量，这个问题的答案要等到用户在生产中撞上回归才会知道。

Shukla et al. (2025) quantify this risk: iterative LLM code refinement paradoxically degrades security, with vulnerabilities rising from 2.1 per sample in early iterations to 6.2 by iterations 8–10 — a 37.6% increase in critical vulnerabilities after just five iterations [9]. He et al. (2025) corroborate the drift risk from a different angle: Cursor adoption produces persistent quality degradation (+30% static analysis warnings, +42% cognitive complexity) that outlasts the transient velocity gains [11].

Shukla 等（2025）量化了这个风险：迭代式的 LLM 代码精化反而会降低安全性，漏洞数从早期迭代每个样本 2.1 个升到第 8-10 次迭代的 6.2 个——仅五次迭代后关键漏洞就增加了 37.6% [9]。He 等（2025）从另一个角度佐证了漂移风险：采用 Cursor 会带来持续的质量退化（静态分析告警 +30%、认知复杂度 +42%），其持续时间超出短暂的速度红利 [11]。

The Kitchen Loop treats regression control as a first-class concern, not an afterthought. Every iteration ends with a regression phase that answers: "Is the system at least as good as it was before this iteration?"

厨房循环把回归控制视为一等公民，而非事后补救。每次迭代以一个回归阶段收尾，回答的问题是："系统现在至少不比这次迭代之前差吗？"

7.2 The Regression Oracle

7.2 回归神谕

Each product domain requires a regression oracle — a repeatable test that answers "is the system still working?" in bounded time. The oracle's properties: Deterministic (same inputs produce same pass/fail on same codebase); Comprehensive (covers the product's critical paths); Fast enough (must complete within the iteration budget); Independent of the loop (the oracle tests the product, not the loop's own output).

每个产品域都需要一个回归神谕（regression oracle）——一个可重复执行的测试，在有限时间内回答"系统还能用吗？"。神谕的属性：确定性（同一代码库下相同输入产出相同的通过/失败）、全面（覆盖产品的关键路径）、够快（必须在迭代预算内完成）、与循环独立（神谕测的是产品，不是循环——它不知道也不在乎是哪个 PR 修了什么）。

7.3 Blocked Combos 注册表

8 System Architecture

8 系统架构

8.1 The Six-Phase Loop

8.1 六阶段循环

The Kitchen Loop is orchestrated by a shell framework that manages state transitions, git worktree isolation, and error recovery. The core logic is delegated to specialized AI skills, each encoding a repeatable, autonomous workflow.

厨房循环由一个 shell 框架编排，它管理状态转换、git worktree 隔离和错误恢复。核心逻辑被委派给专用的 AI 技能（skills），每个技能编码一个可重复、自治的工作流。

Backlog (∼15 min): Evaluates urgency and coverage gaps, promotes candidates to the work queue, generates new scenario tickets when supply runs low.

Backlog（约 15 分钟）：评估紧迫性和覆盖间隙，把候选项推进到工作队列，在供给不足时生成新的场景票。

Ideate (∼15-45 min): Selects a scenario, implements it as a real user would, runs it against the test environment, documents what breaks. Optionally passes through an external feasibility check.

Ideate（约 15-45 分钟）：选取一个场景，像真实用户那样把它实施，在测试环境里跑，记录什么坏了。可选地经一道外部可行性检查。

Triage (∼5-10 min): Converts findings into labeled, prioritized tickets with root cause hypotheses and file pointers. Deduplicates against existing tickets, and reopens tickets whose prior fix PRs were closed without merging.

Triage（约 5-10 分钟）：把发现转成有标签、有优先级的票，附上根因假设和文件指针。与已有票去重，并重开那些先前修复 PR 没被合并就关掉了的票。

Execute (∼30-60 min): For each top-N urgent ticket: creates a feature branch in an isolated worktree, implements the fix, writes tests, opens a PR. Includes backpressure control.

Execute（约 30-60 分钟）：对前 N 张紧急票各自：在隔离的 worktree 里建特性分支、实现修复、写测试、开 PR。包含回压控制。

Polish (∼10-90 min): PR hardening and merging through a graduated state machine. Each PR is tracked with an attempt counter; after a configurable number of failures (default: 1), it is labeled needs-attention and excluded from future processing. Operators can raise the threshold to enable multi-attempt escalation with follow-up ticket creation and failure classification. PRs blocked by security or architectural concerns are retired — closed with a comment routing the ticket back to the backlog. Before any merge, a TOCTOU-safe deletion check verifies no files were unexpectedly removed.

Polish（约 10-90 分钟）：通过一个分级状态机把 PR 加固并合并。每个 PR 配一个尝试计数器；在可配置次数失败后（默认 1 次），打上 needs-attention 标签并排除在后续处理之外。操作员可以提高阈值，启用多次尝试升级机制——伴随后续票创建和失败分类。因安全或架构关切被阻塞的 PR 会被退役——关闭时附一条注释，把工单路由回 backlog。任何合并之前都会做一次 TOCTOU 安全的删除检查，确认没有文件被意外移除。

Regress (∼40-150 min): Runs the regression oracle (read-only — no fixes). Updates loop state directly to the base branch. Promotes confirmed patterns to durable memory. Produces iteration summary with drift metrics. Before updating shared state files (e.g., loop-state.md), the Regress phase re-reads the current version to prevent stale overwrites from interrupted or concurrent runs.

Regress（约 40-150 分钟）：跑回归神谕（只读——不修任何东西）。直接把循环状态更新到基线分支。把已确证的模式提升到持久记忆。产出带漂移指标的迭代摘要。在更新共享状态文件（如 loop-state.md）之前，Regress 阶段会重新读取当前版本，避免被中断或并发的运行用过期内容覆盖。

8.5 The Bar: Production-Readiness Standard

8.5 The Bar：生产就绪标准

The Kitchen Loop applies "The Bar" — a project-specific quality standard defined in a customizable quality.bar_file. The framework ships a generic template covering code quality, PR standards, safety, and documentation; each deployment customizes for its domain. This approach is supported by Abtahi and Azim (2025), who find that categorizing quality issues by type achieves 100% resolution vs. 71.6% for bulk processing [5] — the Kitchen Loop's phase-based architecture naturally decomposes enforcement into categories (functional correctness in Execute, style in Polish, regression safety in Regress).

厨房循环执行 "The Bar"——一份项目特定的质量标准，定义在一个可定制的 quality.bar_file 中。框架自带一个通用模板，覆盖代码质量、PR 标准、安全、文档；每个部署再为自己的领域定制。这一做法得到 Abtahi 和 Azim（2025）的支持，他们发现按问题类型分类处理质量问题可达 100% 解决率，对比批量处理仅 71.6% [5]——厨房循环的阶段式架构天然把执行分解成类别（Execute 管功能正确性、Polish 管风格、Regress 管回归安全）。

For example, the DeFi strategy framework (Case Study A) instantiates The Bar as five domain-specific principles: (1) Production Ready — no TODOs, shortcuts, or "we'll fix later." Ship-quality on merge. (2) Zero Hardcoding — all configuration from resolvers and registries, never literals. (3) Million-Dollar Safe — every external call handled, amounts rounded safely, no silent fallbacks. (4) Hedge-Fund Serious — precise arithmetic everywhere, meaningful test assertions, conservation checks in all failure paths. (5) UX First, Safety Always — clean, declarative APIs; non-bypassable safety gates; secrets redacted from logs.

举例，DeFi 策略框架（案例研究 A）把 The Bar 实例化为五条领域特定原则：(1) Production Ready——没有 TODO、捷径或"以后再修"。合并即出货质量。(2) Zero Hardcoding——所有配置来自 resolver 和 registry，绝不写字面量。(3) Million-Dollar Safe——每个外部调用都处理、金额安全取整、没有静默回退。(4) Style is Sacred——风格约定在代码库里有审美意义；偏离就是债。(5) Modular & Composable——具体策略由可重用 piece 组合而成，不是从头编排。

These principles are enforced through multi-model review tribunals — every PR is evaluated against them before merge. Other domains would define their own principles (e.g., HIPAA compliance for healthcare, WCAG conformance for web accessibility).

这些原则通过多模型评审 tribunal 强制——每个 PR 在合并前都要据此被评估。其他领域会定义自己的原则（如医疗的 HIPAA 合规、Web 无障碍的 WCAG 一致性）。

9 The Discussion Manager: Structured Multi-AI Deliberation

9 Discussion Manager：结构化的多 AI 审议

9.1 Beyond Review Tribunals

9.1 超越评审 tribunal

Section 4.6 introduced multi-model review tribunals — three independent AI reviewers evaluating the same artifact in parallel. The Discussion Manager extends this concept into a full structured deliberation system: a multi-round, multi-model debate framework where heterogeneous AI agents argue substantive positions, challenge each other's reasoning, and converge toward actionable decisions through an impartial moderation layer.

§4.6 引入了多模型评审 tribunal——三位独立的 AI 评审者并行评估同一件 artifact。Discussion Manager 把这个概念扩展为一个完整的结构化审议系统：一个多轮、多模型的辩论框架，异质的 AI 智能体在其中辩护实质性立场、相互挑战推理、并经一个中立的仲裁层趋同到可执行的决策上。

Where review tribunals answer "is this code correct?", the Discussion Manager answers harder questions: "should we build this at all?", "which of three competing architectures best serves the specification surface?", and "what are we not seeing?" These are the judgment-intensive decisions that the Kitchen Loop encounters in its Ideate and Triage phases — decisions where a single model's blind spots can compound into wasted iterations.

评审 tribunal 回答的是"这段代码对吗？"，Discussion Manager 回答的是更难的问题："这东西到底要不要做？"、"在三种竞争架构里哪个最贴合规约面？"、"我们漏看了什么？"——这些都是判断密集型决策，是厨房循环在 Ideate 和 Triage 阶段会撞到的，单个模型的盲点会在这里被放大成被浪费的迭代。

The Discussion Manager operationalizes what Cai et al. (2025) identify as the "Debate-Based Cooperation" pattern — adversarial argumentation to surface errors — which appears in only 4 of 94 surveyed multi-agent systems (4.3% adoption) [8]. The low adoption rate, despite theoretical value for error discovery, suggests this is an underexplored design space. The Kitchen Loop's production implementation, with its anti-sycophancy safeguards and empirical validation across 23 discussions, represents one of the first operational deployments of this pattern at scale.

Discussion Manager 把 Cai 等（2025）所刻画的"基于辩论的协作"模式（Debate-Based Cooperation）——以对抗式论证暴露错误——付诸生产实践。该模式在 Cai 等调查的 94 个多智能体系统中只有 4 个采用（4.3%）[8]。尽管在错误发现上有理论价值，采用率却如此之低，提示这是一个尚未充分探索的设计空间。厨房循环的生产实现，配上反阿谀（anti-sycophancy）保护和 23 次讨论的实证验证，代表了对这一空间一次有据可循的探索。

9.2 Architecture

9.2 架构

The Discussion Manager uses a centralized moderator architecture with an information firewall between moderator and debaters:

Discussion Manager 采用集中式仲裁者架构，仲裁者与辩论者之间有一道信息防火墙：

Key design decisions: Heterogeneous models — three different model families (Gemini, Codex/GPT, Claude) ensure genuine perspective diversity. Each model brings different training data, different reasoning patterns, and different failure modes. Homogeneous debate (three instances of the same model) rapidly converges to groupthink. Centralized moderation — the moderator orchestrates turn order, checks convergence, and writes the synthesis — but never injects opinions into the debate itself. Centralized judge structures outperform peer-review structures for debate quality (Yao et al., 2025). Claude subagent isolation — when the moderator is Claude, the Claude debater runs as an isolated subagent with no access to the moderator's reasoning, notes, or context. The subagent sees only the debate prompt. This is the information firewall that prevents the moderator's framing from contaminating the debate. Code grounding — for technical discussions, debaters receive codebase context (specific files, architecture documents, or topic-aware summaries). Gemini gets autonomous file-search access; other debaters get relevant content injected into their prompts. This anchors debate in verifiable facts rather than rhetorical polish.

关键设计决策：异质模型——三个不同的模型家族（Gemini、Codex/GPT、Claude）确保观点的真正多样性。每个模型带来不同的训练数据、不同的推理模式、不同的失败模式。同质辩论（同一模型的三个实例）会迅速收敛到群体思维。集中仲裁——仲裁者编排发言顺序、检查收敛、并撰写综述——但绝不在辩论本身中注入意见。这就是信息防火墙：仲裁者看辩论者的内容，反过来不行。轮次上限 2-3——长辩论会退化成阿谀式收敛（"你说得对，我重新考虑了"）；短辩论保留有产出的分歧。结构化的提议-质询-反驳——每位辩论者必须首先以书面立论；只有看到对方立论后才能提出质询；反驳必须直接对应质询。这强迫深度而非平行独白。

9.4 Implementation and Open Challenges

9.4 实现与未决挑战

The Discussion Manager is implemented as a Python orchestrator (discuss.py, open-sourced in the companion repository) that manages conversation creation, turn execution, convergence checking, ratification, and report generation. It is designed for use at three integration points: (1) Ideate phase for architectural decisions, (2) Code review in audit mode, and (3) Retrospective comparing implementation against agreed designs. In the current open-source release, the Discussion Manager is invoked manually; automated orchestrator integration is planned.

Discussion Manager 实现为一个 Python 编排器（discuss.py，已在配套仓库开源），管理对话创建、轮次执行、收敛检查、批准与报告生成。它设计在三个集成点使用：(1) Ideate 阶段的架构决策，(2) 审计模式下的代码评审，(3) 对照已商定设计的实现回顾。在当前开源发行中，Discussion Manager 的调用是手动的（操作员触发），而非完全自治；与 Backlog/Triage 的全自动集成是一个开放工程问题，需要为何时辩论值不值得算力做出可靠判断。

Despite the epistemic improvements, several challenges remain: Cost — three models for 3-5 rounds at ∼400 words per turn consumes 15,000-25,000 tokens per model per discussion. With the ∼50% round efficiency observed in the corpus, approximately 40% of token spend is on convergence dynamics rather than novel insight. The cost implications for scaling are discussed in Section 11.4. Role calcification — even with heterogeneous models, each model tends toward a fixed role (Gemini as breadth analyst, Codex as bug hunter, Claude as compromise broker). Random role assignment per discussion can help, but the underlying model tendencies persist. Evaluation circularity — when the moderator is Claude and one meta-analyst is also Claude, self-assessment bias is difficult to fully eliminate. The independent Codex meta-analysis mitigates this but does not resolve it entirely. Scaling beyond three — the current protocol is optimized for three debaters. Scaling to more participants increases round duration and convergence complexity without proportional improvement in perspective diversity.

尽管在认识论上有所改进，仍有若干挑战：成本——3 个模型、3-5 轮、每轮约 400 字，每个模型每次讨论消耗 15,000-25,000 token。在语料中观察到约 50% 的轮次效率，这意味着大约 40% 的 token 花在收敛动力学上而非新洞察上。规模化的成本影响在 §11.4 讨论。角色僵化——即便用异质模型，每个模型都倾向固定角色（Gemini 提广度、Codex 提技术深度、Claude 综合）。这造成可预测的辩论几何，限制了观点真正的随机性。收敛 vs. 完备——硬轮次上限（默认 3）阻止阿谀式收敛，但也可能在分歧确实需要更多审议时切短了讨论。当前实现缺乏区分"我们已收敛"与"我们用完了轮次"的机制。

These are active research questions. The framework's modular design — separating orchestration, moderation, and debater execution — allows incremental improvement without architectural overhaul.

这些都是活跃的研究问题。框架的模块化设计——把编排、仲裁、辩论者执行分开——允许增量改进而无需架构大手术。

10.2 The Regression Oracle: Fork Execution

10.2 回归神谕：Fork 执行

The regression oracle is a suite of demo strategies executed on Anvil (a local EVM chain fork). Each strategy runs against a fork of the target chain, compiles intents to transactions, executes them, and verifies on-chain state changes.

回归神谕是一组在 Anvil（本地 EVM 链 fork）上执行的演示策略。每个策略针对目标链的一个 fork 运行，把意图编译成交易、执行，并校验链上状态变化。

10.5 The Self-Healing Property

10.5 自愈属性

The loop discovered and fixed its own infrastructure bugs through its standard process — no human diagnosis required: Apple Silicon memory bug — PR merge automation used the wrong memory page size (4096 instead of 16384), causing available memory to read 4x too low. This blocked merging for 5 consecutive iterations. The loop-review skill identified the pattern; Execute fixed it; Regress confirmed the fix. Codex feasibility checker — the external AI cooperation model (Section 12.3) matured through the same loop cycle — strict response contracts, timeout handling, and salvage paths were all refined by the loop itself. Ideate persistence bug (iter 13-19) — six iterations of lost deliverables (reports not committed to git) before the loop identified the pattern, filed a ticket, and shipped a fix that decoupled the commit/push lifecycle. Verified stable from iter 19 onward. PR Manager death spiral (iter 71-73) — permanent needs-attention labels caused a 62-hour deadlock with zero merges. The loop review diagnosed the root cause; subsequent iterations added TTL-based label recovery and a circuit breaker. Full recovery by iter 74 (5/5 PRs merged). Demo regression blind spot (iter 53-74) — twenty iterations of blind demo testing because ALCHEMY_API_KEY was missing in worktree environments. The loop detected the pattern, shipped a .env copy to worktrees with a pre-flight key check, and confirmed the fix in iter 75.

本节刻意保持坦诚。循环并非绝无差错。但其失败模式属于运维性质（合并自动化 bug、API 密钥管理），而非安全关键性质（数据丢失、资金损失、生产中断）。架构隔离确保运维故障浪费的是迭代时间，而不是用户信任。

13.4 Human-in-the-Loop Cost Constraints

13.4 Human-in-the-Loop 的成本约束

The loop is autonomous end-to-end: it generates scenarios from the specification surface, fills its own backlog, triages findings into tickets, implements fixes, and auto-merges PRs after multi-model review and CI. No human is in the critical path during normal operation. The human role reduces to (1) initial specification surface definition, (2) occasional strategic steering, and (3) supervisory intervention when the loop encounters failure modes it cannot self-correct.

循环是端到端自治的：它从规约面生成场景，自行填充待办列表，将发现分诊为工单，实现修复，并在多模型评审和 CI 通过后自动合并 PR。正常运行期间没有人类处于关键路径上。人类的角色缩减为：(1) 初始规约面定义，(2) 偶尔的战略转向，(3) 当循环遇到自己无法自纠的失败模式时的监管介入。

In practice, the most common intervention trigger was merge conflict resolution gone wrong: LLMs resolving git conflicts would sometimes silently undo work from previous commits or PRs, requiring a human to notice the regression and revert. GitHub infrastructure issues (API rate limits, webhook failures, stale branch state) were the second most frequent cause. Section 12.2 frames the steady-state human cost as ∼30-60 minutes per week once calibrated — but this assumes the loop is running smoothly. Intervention spikes during infrastructure instability.

实践中最常见的介入触发是合并冲突解决出错：LLM 在解决 git 冲突时偶尔会静默撤销此前提交或 PR 的工作，需要人工注意到回归并回退。GitHub 基础设施问题（API 速率限制、webhook 故障、陈旧分支状态）是第二大常见原因。第 12.2 节把稳态人力成本定为校准后每周约 30-60 分钟——但这个数字假设循环运行顺畅。当基础设施问题堆积时，介入成本会显著跃升，直到循环的自我改进系统识别该模式并自动修复。

The practical scaling constraint is environmental: test environment throughput (Anvil fork startup time, API rate limits), CI pipeline capacity, and the cost of multi-model review tokens per PR. Drain mode (Section 5.5) throttles output when PR backpressure exceeds a threshold, ensuring the loop does not outrun its own merge infrastructure.

实际的可扩展性约束是环境性的：测试环境吞吐（Anvil fork 启动时间、API 速率限制）、CI 流水线容量，以及每个 PR 多模型评审的 token 成本。Drain 模式（第 5.5 节）在 PR 背压超过阈值时节流输出，确保循环不会跑在合并基础设施的前面。

The Discussion Manager adds a separate cost dimension. Three models debating for 3-5 rounds at ∼400 words per turn consumes 15,000-25,000 tokens per model per discussion. With the ∼50% round efficiency observed in our 23-discussion corpus (Section 7.3), approximately 40% of deliberation token spend produces convergence dynamics rather than novel insight. At current subscription pricing this cost is absorbed into flat-rate plans, but metered API usage would make frequent deliberation expensive. Per-iteration compute cost breakdown beyond the aggregate $0.38/PR figure has not yet been instrumented — this is a gap we intend to address in future work.

Discussion Manager 增加了另一个维度的成本。三个模型辩论 3-5 轮、每轮约 400 词，单次讨论每个模型消耗 15,000-25,000 token。在我们 23 场讨论的语料中观察到约 50% 的轮次效率（§7.3），意味着约 40% 的审议 token 支出产生的是收敛动态而非新洞见。按当前订阅定价这部分成本被吸收进套餐，但按量计费的 API 用法会让频繁审议变得昂贵。超出合计 $0.38/PR 之外的每次迭代计算成本分解目前尚未做仪表化——这是我们计划在后续工作中补齐的缺口。

These constraints do not invalidate the framework, but they bound its applicability: the Kitchen Loop is most efficient in domains where (a) the specification surface is well-defined upfront, (b) the regression oracle provides high-confidence automated verification, and (c) the human's role can be genuinely asynchronous rather than synchronous with each iteration.

这些约束并未否定框架，但圈定了其适用范围：Kitchen Loop 在以下领域最为高效——(a) 规约面前期定义清晰，(b) 回归 oracle 提供高置信度的自动化验证，(c) 人类角色能真正做到异步而非与每次迭代同步。

13.5 Open Problems

13.5 开放问题

Our deployments expose four open problems that we believe warrant dedicated research: OP1: Oracle Transfer. The Kitchen Loop relies on a bespoke regression oracle per domain (chain fork execution for DeFi, quality-gate pipeline for signals). Automatic generation of regression oracles from natural-language specifications — eliminating the per-domain engineering cost — is an unsolved challenge. OP2: Specification Acquisition. Our method assumes an enumerable specification surface. For legacy codebases with implicit specifications, automating surface extraction from telemetry, documentation, and user behavior is a critical bottleneck for adoption. OP3: Multi-Objective Drift. Current drift metrics focus on functional correctness. Extending the framework to simultaneously monitor non-functional requirements (latency, security, fairness) without human intervention remains open. Our drift detection (Section 5.5) would need to compose multiple objective functions without one dominating the pause-gate signal. OP4: Sycophancy at Scale. The Discussion Manager mitigates sycophancy in 3-model debates (Section 7), but optimal model composition and debate protocols for larger, heterogeneous agent swarms are unknown. Our corpus of 23 discussions is too small to establish whether the observed SS < 20 threshold generalizes (cf. Yao et al., 2025 [14]).

我们的部署暴露出四个我们认为值得专门研究的开放问题：OP1：Oracle 迁移。 Kitchen Loop 依赖领域定制的回归 oracle（DeFi 用链上 fork 执行，信号用质量门流水线）。从自然语言规约自动生成回归 oracle——以消除领域工程成本——仍是未解难题。OP2：规约获取。 我们的方法假设规约面可枚举。对于规约隐式的遗留代码库，从 telemetry、文档和用户行为中自动提取规约面是采纳的关键瓶颈。OP3：多目标漂移。 当前漂移指标聚焦功能正确性。将框架扩展为同时监测非功能需求（延迟、安全、公平性）而无需人工介入仍是开放问题。我们的漂移检测（第 5.5 节）需要组合多个目标函数且无任何一个主导 pause-gate 信号。OP4：规模化的奉承倾向。 Discussion Manager 在 3 模型辩论中缓解了 sycophancy（第 7 节），但对于更大、更异构的 agent 群体，最优模型组合与辩论协议尚不清楚。我们 23 场讨论的语料太小，无法确认观察到的 SS < 20 阈值是否可泛化（cf. Yao et al., 2025 [14]）。

13.6 When the Kitchen Loop Should NOT Be Used

13.6 何时不应使用 Kitchen Loop

The framework is not universally applicable. It should not be applied when: (1) ground truth is weak or unobservable — without a reliable oracle, the verification layer provides false confidence; (2) success criteria are highly subjective (aesthetic quality, UX taste) — the oracle cannot arbitrate matters of judgment; (3) the specification surface is not enumerable, as in exploratory R&D where the goal is discovery rather than convergence; or (4) safety-critical domains lack robust external verification — the oracle's bounded coverage (Section 4.1) is insufficient when failures have irreversible consequences.

该框架并非放之四海皆准。不应在以下情况下使用：(1) 事实之源薄弱或不可观测——没有可靠 oracle，验证层只会提供虚假信心；(2) 成功标准高度主观（美学质量、UX 品味）——oracle 无法裁决审美判断；(3) 规约面不可枚举，例如以探索为目标而非收敛的探索性研发；(4) 安全关键领域缺乏稳健的外部验证基础设施——循环只能和其 oracle 一样可靠，在安全关键场景里这个标准极高。

17 Conclusion

17 结论

17.1 The Shift

17.1 转变

In our observation, software engineering is undergoing a phase transition. Code production — once the bottleneck — is becoming a commodity. Code review — once a human-only activity — is becoming automated. The competitive advantage is shifting to: (1) Specification — knowing what to build (the AaU1000 method); (2) Verification — proving it works (unbeatable tests); (3) Convergence — ensuring it keeps working (regression and drift control).

在我们观察中，软件工程正经历一次相变。代码生产——曾经的瓶颈——正变成 commodity。代码评审——曾经仅限人类的活动——正变得自动化。竞争优势正转移至：(1) 规约——知道要造什么（AaU1000 方法）；(2) 验证——证明它能工作（unbeatable tests）；(3) 收敛——确保它持续工作（回归与漂移控制）。

17.2 The Evidence

17.2 证据

Across two production systems and 285+ combined iterations (Sections 8.3 and 9.3), zero regressions were detected by the regression oracle, quality gates improved monotonically from 76–91% to 100%, and the loop autonomously fixed 17+ infrastructure bugs in its own tooling. The second system validated portability: the same architecture applied to a fundamentally different domain achieved comparable results with a 25x faster iteration cycle, requiring only two reimplemented components (specification surface and regression oracle). The unified trust model held under sustained autonomous operation without increasing human intervention.

跨两个生产系统、合计 285+ 次迭代（第 8.3 与 9.3 节），回归 oracle 检测到的回归数为零，质量门从 76–91% 单调改进到 100%，循环自治地修复了自身工具里 17+ 个基础设施 bug。第二个系统验证了可移植性：同一架构应用于根本不同的领域取得了可比结果，迭代周期快 25 倍，仅需重新实现两个组件（规约面与回归 oracle）。统一信任模型在持续自治运行下未增加人类介入。

17.3 The Invitation

17.3 邀请

The Kitchen Loop is not the future of software development. It is a practice available today, on a codebase of any size, with tools that already exist. The prerequisites are: (1) an enumerable specification surface — what does your product claim to do? (2) an automatable test environment — can an AI agent exercise your product? (3) a regression oracle — can you answer "is the system still working?" in bounded time? (4) the discipline to run it continuously — and trust the output when the tests are unbeatable.

Kitchen Loop 不是软件开发的未来。它是今天、在任何规模的代码库上、用已存在的工具就可践行的方法。前提是：(1) 规约面可枚举——你的产品自称做什么？(2) 测试环境可自动化——AI agent 能否操练你的产品？(3) 回归 oracle——你能否在有界时间内回答"系统是否仍在正常工作？"(4) 持续运行它的纪律——并在测试 unbeatable 时信任其输出。

In our experience, the tests are the trust layer. The spec is the compass. The loop is the engine. Given sufficient verification infrastructure, the product evolves itself.

按我们的经验，测试是信任层。规约是罗盘。循环是引擎。给予足够的验证基础设施，产品便会自我演化。

17.4 Testable Hypotheses

17.4 可检验的假设

Our deployments suggest four empirical hypotheses that subsequent work could validate, replicate, or refute: H1: Coverage-exhaustion systems (as defined in the regime taxonomy, Section 2.9) discover more user-visible failures per iteration than task-completion systems in partially mature products with >50% specification surface coverage. H2: Adversarial UAT gates — sealed test cards executed by a fresh evaluator with zero implementation context — reduce false-positive readiness assessments compared with implementer-authored test suites alone. H3: Tier-weighted scenario selection (Foundation 30% / Composition 50% / Frontier 20%) discovers more bugs per iteration than uniform random selection across the specification surface. The superlinear growth of the Composition tier (Section 3.6) predicts that the advantage increases with product maturity. H4: In user-facing verification tasks, weak-model evaluation (the least capable model available) is a better proxy for real-user verifiability than strong-model evaluation, because weak models fail on the same ambiguities that trip real users.

我们的部署提示出四个经验性假设，供后续工作验证、复现或反驳：H1： 在规约面覆盖率 >50% 的部分成熟产品中，覆盖穷尽型系统（如第 2.9 节制度分类所定义）比任务完成型系统每次迭代发现更多用户可见故障。H2： 对抗性 UAT 门——由一位零实现上下文的新评估员执行的密封测试卡——相较于实现者自撰测试套件，能降低就绪度评估的假阳率。H3： 分层加权场景选择（Foundation 30% / Composition 50% / Frontier 20%）比在规约面上均匀随机选择每次迭代发现更多 bug。Composition 层的超线性增长（第 3.6 节）预测该优势会随产品成熟度增大。H4： 在用户面向的验证任务中，弱模型评估（可用中能力最差的模型）比强模型评估是真实用户可验证性的更好代理，因为弱模型会在绊倒真实用户的同类歧义上失败。

These hypotheses are intended to facilitate direct replication, extension, or refutation in future agentic systems research.

这些假设旨在便于在未来 agentic 系统研究中进行直接复现、扩展或反驳。

17.5 Known Limitations & Future Work

17.5 已知局限与未来工作

The Kitchen Loop has five structural limitations that bound its applicability: (1) Single-threaded execution — the current orchestrator runs one iteration at a time. Parallelization across multiple worktrees is architecturally straightforward but not yet implemented (Section 13.3). (2) Enumerable specification surface required — the method assumes the product's capabilities can be listed as a coverage matrix. Legacy monoliths with implicit specifications require a specification-extraction step (OP2, Section 13.5) before the loop can operate. (3) Oracle quality is the ceiling — the loop can only catch what the regression oracle can verify. If the oracle misses a failure mode, the loop is blind to it. Oracle transfer across domains (OP1) remains an open research problem. (4) Human role persists — backlog grooming, specification design, and production promotion require human judgment. At scale, human merge capacity — not AI generation speed — becomes the binding constraint (Section 13.4). (5) Two-domain validation only — our evidence spans two production systems in the DeFi domain. Generalization to other domains (Section 16) is architecturally supported but empirically unvalidated.

Kitchen Loop 有五项结构性局限圈定其适用范围：(1) 单线程执行——当前编排器每次只运行一次迭代。跨多 worktree 并行化在架构上直接可行，但尚未实现（第 13.3 节）。(2) 需要可枚举的规约面——该方法假设产品能力可作为覆盖矩阵列出。具有隐式规约的遗留单体代码需要一个规约抽取步骤（OP2，第 13.5 节）后循环才能运作。(3) Oracle 质量即上限——循环只能捕获回归 oracle 能验证的东西。若 oracle 漏掉某种失败模式，循环对它是盲的。跨领域的 oracle 迁移（OP1）仍是开放研究问题。(4) 人类角色仍然存在——待办管理、规约设计、生产发布需要人类判断。到达一定规模时，人类合并能力——而非 AI 生成速度——成为约束（第 13.4 节）。(5) 仅两个领域的验证——我们的证据涵盖 DeFi 领域两套生产系统。向其他领域的泛化（第 16 节）在架构上已支持但经验上尚未验证。

C.2 E.2 Anti-Signal Canary Details

C.2 E.2 反信号 Canary 细节

Tier 1 — Obviously Bad (6 canaries): fabricated event for non-existent entity; signal with impossible metric value (e.g., 999,999% yield); false report for an entity that was never affected; signal with null/empty identifiers; signal with empty title and description; signal with out-of-range confidence score. Result: 100% caught across 163 iterations. Zero escapes.

Tier 1——显然错（6 个 canary）： 为不存在的实体编造事件；不可能的指标值（例如 999,999% 收益率）；从未受影响实体的伪报；带空/null 标识符的信号；标题和描述皆空的信号；置信分超出范围的信号。结果： 163 次迭代中 100% 捕获，零漏网。

Tier 2 — Shadow (5 canaries): real event that already concluded; known trend reported weeks ago (stale); opportunity that dropped below threshold since detection; valid signal for a deprecated/discontinued entity; accurate data from a non-authoritative source. Result: Initially 33% caught by L1-L3 (iter 1). Now 100% (iter 163) — improved as verifiers were added and L2 factual checkers gained cross-reference capabilities.

Tier 2——阴影（5 个 canary）： 已结束的真实事件；数周前已报过的已知趋势（陈旧）；检测后已跌破阈值的机会；已弃用/下线实体的有效信号；非权威来源的准确数据。结果： L1-L3 初始捕获 33%（迭代 1）。如今 100%（迭代 163）——随着 verifier 增补、L2 事实核查获得交叉参照能力而提升。

Tier 3 — Adversarial (5 canaries): real data + fabricated interpretation; accurate raw data + incorrect derived conclusion; valid input data + wrong calculation; correct methodology applied to wrong time window; signal mixing real and fabricated sub-claims. Result: Initially 67% caught by L1-L3 (iter 1). Now 100% (iter 163) — the remaining required cross-referencing conclusions against source data, which improved as verifier coverage reached 100%.

Tier 3——对抗性（5 个 canary）： 真实数据 + 编造的解读；准确原始数据 + 错误派生结论；有效输入 + 错误计算；正确方法论应用于错误时间窗口；掺杂真实与编造子断言的信号。结果： L1-L3 初始捕获 67%（迭代 1）。如今 100%（迭代 163）——剩余需要把结论与源数据交叉参照，随着 verifier 覆盖达到 100% 而改善。

Tier 4 — Mixed True/False (5 canaries, added ∼iter 100): signal with 3 true claims and 2 fabricated claims blended together; accurate quantitative data with fabricated qualitative assessment; real protocol event attributed to wrong protocol; valid historical data presented as current; correct analysis of real data with one inverted conclusion. Result: 100% caught. Tests the quality gate's ability to detect partial failures rather than binary pass/fail.

Tier 4——真假混合（5 个 canary，约在迭代 100 时加入）： 混合了 3 条真实与 2 条编造断言的信号；精确定量数据 + 编造的定性评估；将真实协议事件归因到错误协议；把历史数据当作当期呈现；对真实数据的正确分析但结论倒置。结果： 100% 捕获。测试质量门检测部分失败（而非二元通过/失败）的能力。

API Degradation Canaries (3 canaries): simulated timeout from primary data source; simulated error response from price oracle; simulated partial data return (50% of expected fields). Result: 100% resilience (3/3). Quality gates degrade gracefully — they flag signals as unverifiable rather than producing false passes when dependencies fail.

API 退化 Canary（3 个 canary）： 模拟主数据源超时；模拟价格 oracle 的错误响应；模拟部分数据返回（50% 的预期字段）。结果： 100% 韧性（3/3）。质量门优雅降级——依赖失效时，它把信号标记为不可验证而不是产出假阳 PASS。

C.3 E.3 Key Lesson: Rapid Iteration Reveals Intermittent Bugs

C.3 E.3 关键教训：快速迭代揭示间歇性 Bug

One agent's validation failure rate fluctuated between 75-100% across iterations because its data source API returns different formats at different times. A single test run would show "pass" or "fail" — 163 iterations reveals the ∼85% steady-state rate. This class of bug, invisible to traditional CI, demonstrates the value of coverage through repetition, not just breadth.

一个 agent 的验证失败率在迭代之间在 75-100% 之间波动，因为其数据源 API 在不同时间返回不同格式。单次测试运行只会显示"pass"或"fail"——163 次迭代则揭示出约 85% 的稳态失败率。这类传统 CI 不可见的 bug 展示了"通过重复实现覆盖"的价值，而不只是靠广度。

C.4 E.4 Drift Detection at Scale

C.4 E.4 大规模漂移检测

The drift detection system monitors quality metrics over a sliding window (5 recent vs. 20 baseline iterations). The key insight: drift detection provides early warning before quality gates fail — a 5% drop in factual pass rate over 10 iterations triggers an alert before any individual signal fails hard. This is the mechanism that allows the loop to operate autonomously for 163+ iterations without human supervision.

漂移检测系统在一个滑动窗口（最近 5 次 vs 基线 20 次）上监控质量指标。关键洞察：漂移检测在质量门失败之前提供预警——10 次迭代内事实通过率下降 5% 会触发告警，早于任何单条信号硬失败。这就是让循环在无人工监管下自治运行 163+ 次迭代的机制。
